# 报名表数据模型与导出约束

## 1. 目标

本文件用于补齐“每个比赛配置自己的报名表”在真实数据层的设计，解决以下问题：

- 动态字段如何存储
- 校验规则如何持久化
- 团队报名结构如何建模
- 驳回后补交如何留痕
- 导出字段如何与报名表定义对齐

## 2. 设计原则

- 比赛报名是强业务对象，不做纯前端临时 JSON 拼接。
- 报名表结构要可版本化，不能一改字段就污染历史报名。
- 报名提交要保留快照，不能只保留当前最新值。
- 导出字段顺序要由表单定义驱动，而不是导出时临时拼。
- 团队成员既要支持固定公共字段，也要支持比赛特有扩展字段。

## 3. 总体建模结论

当前采用“关系型主干 + JSONB 扩展 + 版本快照”的混合模型：

- 表单定义：关系表
- 动态校验规则：`JSONB`
- 当前报名值：`JSONB`
- 报名历史快照：`JSONB`
- 团队成员：独立关系表 + 成员扩展 `JSONB`

说明：

- 阶段 2 不采用通用 EAV（`field_value` 一行一值）模型。
- 原因是首版主要需求是提交、审核、导出与留痕，不是任意字段高频统计。
- 后续如果要按自定义字段做复杂统计，再补投影表或搜索索引。

## 4. 核心实体

### 4.1 `registration_form`

表单主记录，按比赛一对一或一对多管理。

建议字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `competition_id` | 所属比赛 |
| `status` | `draft / published / archived` |
| `apply_mode` | `individual / team / both` |
| `max_team_size` | 最大团队人数 |
| `current_version_id` | 当前生效版本 |
| `created_by` | 创建人 |
| `created_at / updated_at` | 时间戳 |

说明：

- 阶段 2 默认每场比赛只有一套生效报名表，但允许存在多个历史版本。

### 4.2 `registration_form_version`

表单版本头信息。

建议字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `form_id` | 所属报名表 |
| `version_no` | 版本号，从 1 开始递增 |
| `status` | `draft / active / retired` |
| `change_note` | 版本说明 |
| `published_at` | 生效时间 |
| `created_by` | 操作人 |

说明：

- 一旦某版本被报名记录引用，就不能原地修改结构化字段定义。
- 后续任何结构变更，都必须创建新版本。

### 4.3 `registration_form_field`

表单字段定义，归属于具体版本。

建议字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `form_version_id` | 所属版本 |
| `field_key` | 机器键名，版本内唯一 |
| `label` | 字段标题 |
| `field_type` | 字段类型 |
| `scope` | `registration / team_member` |
| `is_required` | 是否必填 |
| `display_order` | 展示顺序 |
| `export_order` | 导出顺序 |
| `export_label` | 导出列名 |
| `placeholder` | 占位提示 |
| `help_text` | 帮助文案 |
| `default_value_json` | 默认值 |
| `options_json` | 下拉、单选、多选项 |
| `validation_json` | 校验规则 |
| `visibility_rule_json` | 显隐规则，阶段 2 可先简化 |

### 4.4 `registration`

当前报名主记录，代表“同一报名对象在同一比赛中的当前有效记录”。

建议字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `competition_id` | 所属比赛 |
| `form_version_id` | 首次提交绑定的表单版本 |
| `applicant_user_id` | 报名发起人 |
| `apply_mode` | `individual / team` |
| `status` | 报名状态 |
| `revision_no` | 当前修订号 |
| `payload_json` | 当前报名字段值 |
| `applicant_snapshot_json` | 提交时的申请人基础资料快照 |
| `submitted_at` | 最近一次提交时间 |
| `approved_at` | 审核通过时间 |
| `last_reviewed_at` | 最近审核时间 |
| `last_reviewed_by` | 最近审核人 |
| `latest_review_comment` | 最近审核意见摘要 |
| `created_at / updated_at` | 时间戳 |

说明：

- `payload_json` 以 `field_key -> value` 形式存储当前值。
- 常用筛选维度如学院、专业、年级应同时保留在 `applicant_snapshot_json` 中，必要时可做单独冗余列。

### 4.5 `registration_team_member`

团队成员当前值表。

建议字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `registration_id` | 所属报名 |
| `sort_order` | 排序 |
| `is_leader` | 是否队长 |
| `member_role` | 成员角色 |
| `name` | 姓名 |
| `student_no` | 学号 |
| `college` | 学院 |
| `major` | 专业 |
| `grade` | 年级 |
| `phone` | 手机号 |
| `email` | 邮箱 |
| `member_payload_json` | 团队成员的扩展字段值 |

说明：

- 公共字段独立列，便于筛选和导出。
- 比赛特有的成员扩展字段放在 `member_payload_json`。

### 4.6 `registration_revision`

报名修订快照表，用于解决“驳回后补交留痕”。

建议字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `registration_id` | 所属报名 |
| `revision_no` | 修订号 |
| `action_type` | `save_draft / submit / resubmit / withdraw` |
| `payload_snapshot_json` | 报名字段快照 |
| `team_snapshot_json` | 团队成员快照 |
| `form_version_id` | 当时引用的表单版本 |
| `created_by` | 操作人 |
| `created_at` | 时间 |

说明：

- 阶段 2 采用“同一 `registration` 记录 + 多次 `revision` 快照”的方式。
- 不采用“驳回后生成全新报名记录”的策略，避免一人一赛多条主记录混乱。

### 4.7 `registration_audit_log`

审核状态变化日志表。

建议字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `registration_id` | 所属报名 |
| `from_status` | 原状态 |
| `to_status` | 新状态 |
| `action` | `approve / reject / cancel / withdraw / resubmit` |
| `operator_user_id` | 操作人 |
| `operator_role` | 操作角色 |
| `comment` | 备注 / 驳回原因 |
| `created_at` | 时间 |

## 5. 动态字段存储策略

### 5.1 结论

- 当前值存 `registration.payload_json`
- 历史值存 `registration_revision.payload_snapshot_json`
- 字段定义与校验规则存 `registration_form_field`

### 5.2 示例

```json
{
  "project_name": "智能问答助手",
  "advisor_name": "张老师",
  "contact_phone": "13800000000",
  "source_code_repo": "https://example.com/repo",
  "attachments": [
    {
      "fileKey": "competitions/2026/attachments/demo.pdf",
      "fileName": "demo.pdf"
    }
  ]
}
```

### 5.3 为什么不用纯字段值表

- 首版最核心的操作是提交、回显、审核、导出。
- 这些场景下按整份报名表读写更自然。
- 若现在强上 EAV，会让提交、版本快照、导出拼列变复杂。

## 6. 校验规则持久化

### 6.1 `validation_json` 建议格式

```json
{
  "minLength": 2,
  "maxLength": 100,
  "pattern": "^[A-Za-z0-9_-]+$",
  "minItems": 1,
  "maxItems": 5,
  "accept": [".pdf", ".docx"],
  "maxFileSizeMB": 20
}
```

### 6.2 执行规则

- 后台配置时保存到 `validation_json`
- 前端按元数据生成 `Zod` 校验
- 后端提交时再次按同一规则校验

说明：

- 校验不能只做前端校验，后端必须兜底。

## 7. 团队报名规则

### 7.1 结论

- 团队报名的“队伍本身”存在 `registration`
- 队伍成员存在 `registration_team_member`

### 7.2 基本规则

- 一条团队报名记录只允许一个队长
- `registration.apply_mode = team` 时，成员表至少有 1 条且必须有队长
- 团队成员人数不得超过 `registration_form.max_team_size`
- 若比赛仅支持个人报名，则不允许写入成员表

### 7.3 成员扩展字段

- 团队成员级字段由 `registration_form_field.scope = team_member` 定义
- 实际值写入 `registration_team_member.member_payload_json`

## 8. 版本化规则

### 8.1 表单版本

- 未发布前可编辑 `draft` 版本
- 一旦某版本被提交记录引用，结构性修改必须新建版本
- 新版本生效后，只影响新提交记录

### 8.2 报名补交

- 驳回后补交沿用同一 `registration.id`
- 每次重新提交都会新增一条 `registration_revision`
- `revision_no` 递增

### 8.3 版本绑定规则

- 报名记录默认绑定首次提交时的 `form_version_id`
- 已提交历史记录不自动迁移到新版本
- 如确需迁移，必须由管理员发起显式迁移流程，不能静默改结构

## 9. 重复报名约束

建议业务唯一键：

- 个人报名：`competition_id + applicant_user_id + active_registration`
- 团队报名：`competition_id + leader_user_id + active_registration`

说明：

- `active_registration` 表示状态不处于终态清理之外的有效报名
- 具体终态以 [状态流转矩阵](./状态流转矩阵.md) 为准

阶段 2 的默认业务规则：

- 同一用户对同一比赛只保留一条主报名记录
- 驳回、撤回后如允许再次提交，沿用原记录并提升 `revision_no`

## 10. 导出约束

### 10.1 默认导出形态

- `XLSX`：多 Sheet
- `CSV`：多文件打包下载

### 10.2 默认导出内容

1. `报名汇总`
2. `队员明细`
3. `审核日志`

### 10.3 报名汇总列顺序

固定字段在前，动态字段在后。

固定字段建议顺序：

1. 比赛名称
2. 报名编号
3. 报名状态
4. 报名模式
5. 报名人姓名
6. 学号
7. 学院
8. 专业
9. 年级
10. 联系方式
11. 提交时间
12. 最近审核时间
13. 最近审核意见
14. 修订号

动态字段规则：

- 列名取 `registration_form_field.export_label`
- 顺序取 `registration_form_field.export_order`
- 未配置 `export_label` 时回退到 `label`

### 10.4 队员明细导出

- 一行一个成员
- 必带 `registration_id / competition_id / team_name_or_leader / is_leader`
- 成员扩展字段按 `team_member` 作用域字段顺序追加

### 10.5 编码与兼容

- `CSV` 默认使用 `UTF-8 with BOM`
- `XLSX` 作为后台主导出格式优先支持

### 10.6 大数据量策略

- 小规模导出可同步生成
- 大于阈值时改为分页查询 + 流式写出
- 后续若数据量继续增大，再补异步导出任务表

## 11. 本文解决的问题

本文明确了以下此前悬空的问题：

- 动态字段采用关系定义 + `JSONB` 当前值 / 快照
- 校验规则持久化到 `validation_json`
- 团队成员采用独立表 + 成员扩展 `JSONB`
- 驳回后补交采用“原记录重提 + revision 快照”
- 导出字段由表单版本字段定义驱动
