# 生产数据库迁移与回滚手册

最后更新：2026-05-01

## 1. 目标

用于固化生产环境数据库变更流程，确保：

1. 迁移可重复执行
2. 失败可回滚
3. 变更前后可验证

## 2. 前置条件

1. 已配置生产环境 `DATABASE_URL`
2. 当前发布分支已通过 `npm run lint` 与 `npm run build`
3. 已完成数据库备份（全库或受影响表）
4. 变更窗口内已通知业务方

## 3. 迁移步骤（标准）

1. 在发布分支确认 migration 文件已纳入版本控制（`src/lib/db/migrations/*`）。
2. 执行：

```bash
npm run db:migrate
```

3. 若运行环境受限、无法执行 drizzle migration，再执行兜底 SQL：

```bash
npm run db:migrate:sql
```

4. 迁移后执行健康检查（至少覆盖）：
   - 登录（`/sign-in`）
   - 比赛列表（`/api/competitions`）
   - 报名查询（`/api/me/applications`）
   - 后台报名列表（`/api/admin/applications`）

## 4. 回滚策略

当前项目采用“备份恢复 + 应用回退”双重策略：

1. 若迁移失败且尚未对外放量：
   - 回退应用到上一版本 commit
   - 用发布前备份恢复数据库
2. 若迁移已部分成功：
   - 先停止写入（临时关闭入口或降级只读）
   - 恢复备份
   - 对比恢复前后关键表行数与最近更新时间

## 5. 发布核对清单

1. `AUTH_SECRET` 长度不少于 32。
2. 生产环境禁用 `MVP_ALLOW_ENV_SESSION_FALLBACK`。
3. 上传策略已确认：
   - `UPLOAD_STORAGE_DRIVER=s3`（生产）
   - `S3_*` 参数完整
4. `npm run db:migrate` 成功日志已留档。
5. 端到端回归清单已执行并记录结果（见《端到端回归执行清单》）。

## 6. 风险提示

1. 若继续使用 `UPLOAD_STORAGE_DRIVER=local`，容器重建后附件可能丢失，不可用于生产。
2. `competition_admin` 必须带比赛作用域；无作用域将按 fail-closed 拒绝管理与审核操作。
3. 迁移脚本不替代备份，任何结构变更前必须先备份。
