# 状态流转矩阵

## 1. 目标

本文件用于把“状态枚举”补成“状态机规则”，明确：

- 哪些转换合法
- 由谁触发
- 触发前置条件是什么
- 需要写哪些日志

说明：

- 本文优先覆盖阶段 2 必需的比赛状态与报名状态。
- 问答、通知等内容状态可在后续管理模块落地时再细化。

## 2. 通用原则

- 所有状态变化都必须在服务层执行，不允许前端直接改状态。
- 所有人工审核动作都必须写入日志。
- 缺少“暂停”类状态时，不允许用语义不匹配的状态硬顶替。
- 任何跨状态回退都必须满足前置条件，不能为了“方便操作”绕过业务规则。

## 3. 比赛状态矩阵

当前比赛状态：

- `draft`
- `upcoming`
- `registration_open`
- `in_progress`
- `finished`
- `archived`

### 3.1 合法转换

| From | To | 触发角色 | 前置条件 | 备注 |
|:---|:---|:---|:---|:---|
| `draft` | `upcoming` | `competition_admin` / `super_admin` | 基础信息完整；至少有 1 个可发布表单版本或明确无需报名 | 进入可展示未开报状态 |
| `draft` | `archived` | `competition_admin` / `super_admin` | 草稿废弃；需填写原因 | 用于作废未上线比赛 |
| `upcoming` | `draft` | `competition_admin` / `super_admin` | 尚无已提交报名；未正式开报 | 回退修改基础信息 |
| `upcoming` | `registration_open` | `competition_admin` / `super_admin` / `system` | 报名时间已到；生效表单版本存在 | 可手动开启，也可系统按时间推进 |
| `upcoming` | `archived` | `competition_admin` / `super_admin` | 比赛取消；需同步通知 | 终止上线前比赛 |
| `registration_open` | `in_progress` | `competition_admin` / `super_admin` / `system` | 报名截止；比赛进入进行阶段 | 默认从开报进入进行中 |
| `registration_open` | `archived` | `super_admin` | 发生取消、合并或重大异常；需公告与留痕 | 异常终止 |
| `in_progress` | `finished` | `competition_admin` / `super_admin` / `system` | 比赛流程结束 | 进入已结束 |
| `finished` | `archived` | `competition_admin` / `super_admin` | 导出与资料归档完成 | 进入长期归档 |

### 3.2 不允许的转换

- `registration_open -> upcoming`
- `in_progress -> upcoming`
- `finished -> registration_open`
- `archived -> 其他任意状态`

说明：

- 当前状态集没有“暂停报名”状态，因此不允许把 `registration_open` 回退成 `upcoming` 冒充暂停。
- 如后续确有需求，应新增显式状态，而不是复用旧状态。

### 3.3 比赛状态日志

每次比赛状态变化建议写入 `competition_status_log`，至少包含：

- `competition_id`
- `from_status`
- `to_status`
- `operator_user_id`
- `operator_role`
- `reason`
- `created_at`

## 4. 报名状态矩阵

当前报名状态：

- `draft`
- `submitted`
- `approved`
- `rejected`
- `withdrawn`
- `cancelled`

### 4.1 合法转换

| From | To | 触发角色 | 前置条件 | 备注 |
|:---|:---|:---|:---|:---|
| `draft` | `submitted` | `student_user` | 比赛处于 `registration_open`；校验通过；重复报名校验通过 | 首次提交 |
| `draft` | `cancelled` | `system` / `competition_admin` / `super_admin` | 草稿长期未使用或比赛取消 | 清理无效草稿 |
| `submitted` | `approved` | `competition_admin` / `super_admin` | 操作人有比赛作用域；材料齐全 | 审核通过 |
| `submitted` | `rejected` | `competition_admin` / `super_admin` | 操作人有比赛作用域；必须填写驳回原因 | 审核驳回 |
| `submitted` | `withdrawn` | `student_user` | 比赛仍在报名阶段；尚未被锁定审核 | 学生主动撤回 |
| `submitted` | `cancelled` | `competition_admin` / `super_admin` | 发现违规、比赛取消或无效报名；必须填写原因 | 管理员取消 |
| `rejected` | `submitted` | `student_user` | 比赛仍在报名阶段；已修订并通过校验 | 驳回后补交，`revision_no + 1` |
| `rejected` | `withdrawn` | `student_user` | 学生放弃继续报名 | 主动终止 |
| `rejected` | `cancelled` | `competition_admin` / `super_admin` | 比赛取消或确认不再受理 | 管理员终止 |
| `approved` | `withdrawn` | `student_user` | 比赛尚未进入 `in_progress`；赛事规则允许退出 | 已通过后退出 |
| `approved` | `cancelled` | `competition_admin` / `super_admin` | 资格撤销、比赛取消或发现违规；必须填写原因 | 审批后撤销资格 |
| `withdrawn` | `submitted` | `student_user` | 比赛仍在报名阶段；报名口径允许重新提交 | 沿用原记录重新提交，`revision_no + 1` |
| `withdrawn` | `cancelled` | `competition_admin` / `super_admin` | 管理员确认该记录不再受理 | 终止关闭 |

### 4.2 不允许的转换

- `submitted -> draft`
- `approved -> submitted`
- `approved -> rejected`
- `cancelled -> 其他任意状态`

说明：

- 一旦进入 `submitted`，就不再回到可随意编辑的 `draft`。
- 若已 `approved` 后发现问题，不直接打回 `rejected`，而是走 `cancelled`，保留“曾通过后又撤销”的审计语义。

### 4.3 驳回后补交规则

阶段 2 统一采用：

- 同一条 `registration` 主记录
- 每次补交新增一条 `registration_revision`
- `revision_no` 递增
- 审核日志继续累计，不新建第二条主报名记录

### 4.4 日志要求

以下动作必须写入 `registration_audit_log`：

- 提交
- 驳回
- 通过
- 撤回
- 管理员取消
- 驳回后补交

至少记录：

- `from_status`
- `to_status`
- `operator_user_id`
- `operator_role`
- `comment`
- `created_at`

## 5. 角色与作用域规则

### 5.1 学生

- 只能操作自己的报名记录
- 只能在比赛允许的时间窗内执行提交、撤回、补交

### 5.2 比赛管理员

- 只能审核自己负责的比赛
- 不能跨比赛修改不在自己作用域内的报名状态

### 5.3 全站管理员

- 可跨比赛执行状态修正、取消、归档
- 但异常性动作必须填写原因

## 6. 与时间窗口的关系

时间窗口与状态联动规则如下：

- 比赛未到报名时间，不允许 `draft / rejected / withdrawn -> submitted`
- 比赛结束后，不允许任何新提交、补交或学生撤回
- 比赛进入 `in_progress` 后，默认不再允许学生从 `approved` 转 `withdrawn`

## 7. 与导出和审核的关系

- `approved`、`submitted`、`rejected`、`withdrawn`、`cancelled` 都应可被后台筛选
- 导出时必须同时导出当前状态与最近一次审核意见
- 若需要导出历史过程，应从 `registration_audit_log` 补充

## 8. 本文解决的问题

本文明确了以下此前未冻结的规则：

- 合法状态转换清单
- 每个转换的触发角色
- 每个转换的前置条件
- 驳回后补交采用“原记录重提 + revision 快照”
- 通过后撤销走 `cancelled`，不回打 `rejected`
