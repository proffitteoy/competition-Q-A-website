# 组件选型与分层策略

## 1. 总体原则

组件层按以下顺序分层：

```text
Radix primitives
→ shadcn/ui
→ 页面块组件
→ 业务组件
```

核心约束：

- 基础控件统一从 `shadcn/ui` 体系出
- 不混用 Ant Design、MUI、Element、Mantine 这类完整 UI 套件
- 页面美化组件只能补视觉，不得反向主导基础控件体系
- 复杂交互优先用成熟 headless 库，不自己重写弹窗、表格、拖拽、日历

## 2. 当前仓库里已有的组件源码仓库

当前本地实际存在：

- `ui-main`
- `lucide-main`
- `sonner-main`
- `dnd-kit-main`
- `fullcalendar-main`
- `recharts-main`
- `launch-ui-main`

当前本地不存在：

- `Origin UI`
- `Magic UI`
- `Tremor`
- `Plate`

说明：

- 因此当前项目文档应区分“本地已可参考的源码仓库”和“后续可引入的第三方依赖”。
- 现阶段不应把不存在于仓库的组件库当作已落地前提。

## 3. 各本地仓库的角色定位

### 3.1 `ui-main`

定位：

- `shadcn/ui` 上游源码与模板仓库

对本项目的意义：

- 作为基础 UI 体系和组件清单参考
- 作为 `shadcn` CLI / registry / 组件结构的上游依据

使用原则：

- 把它当“设计系统底座参考源”
- 不把整个 `ui-main` 当作业务项目直接运行时依赖
- 实际项目中应使用拷贝进应用代码库的组件源码，而不是把它当黑盒 UI 包

### 3.2 `lucide-main`

定位：

- 图标源仓库

对本项目的意义：

- 全项目统一图标风格
- 学生端、后台端、状态图标、导航图标都应统一走 `lucide-react`

结论：

- **图标只选 `lucide-react`。**

### 3.3 `sonner-main`

定位：

- 全局 toast 反馈组件

对本项目的意义：

- 统一成功/失败/提醒反馈

适用场景：

- 报名提交成功
- 审核通过/驳回
- 保存失败
- 导出完成
- 上传成功
- 登录状态过期

结论：

- **全局提示统一走 `Sonner`，不在页面里到处散落 Alert。**

### 3.4 `recharts-main`

定位：

- React 图表库源码仓库

对本项目的意义：

- 后台统计图表
- 报名趋势、学院分布、比赛状态等可视化

适用场景：

- 报名趋势折线图
- 学院报名分布柱状图
- 比赛状态饼图
- 年级分布图

结论：

- **图表主库优先用 `Recharts`。**

### 3.5 `dnd-kit-main`

定位：

- 拖拽能力源码仓库

当前观察到的本地事实：

- 本地仓库包结构包含 `@dnd-kit/dom`、`@dnd-kit/react`
- 不包含你当前主模板正在使用的老一代 `@dnd-kit/core`

这意味着：

- 本地 `dnd-kit` 仓库与主模板当前依赖并不是同一代 API 结构

对本项目的意义：

- 报名表字段排序
- FAQ 排序
- 通知排序
- 首页模块排序
- 附件排序

当前建议：

- 第一阶段沿用主模板已带的 `@dnd-kit/core / sortable / utilities`
- 不要因为本地有 `dnd-kit-main` 就立刻升级到另一代 API

### 3.6 `fullcalendar-main`

定位：

- FullCalendar 核心与插件源码仓库

当前观察到的本地事实：

- 本地 `packages` 下有 `core`、`daygrid`、`interaction` 等插件
- 本地仓库里没有 `@fullcalendar/react`

这意味着：

- 本地源码可以帮助理解日历能力与插件边界
- 但真正接入 React 页面时，仍需要实际安装 `@fullcalendar/react`

对本项目的意义：

- 竞赛日历
- 报名截止日历
- 管理员赛程设置

当前建议：

- 第一版可以先做时间线
- 需要日历时再引入 `FullCalendar`

### 3.7 `launch-ui-main`

定位：

- 门户展示组件集合

对本项目的意义：

- 学院竞赛门户首页
- 统计区
- FAQ 区
- CTA 区
- 页脚与展示型导航

说明：

- 在当前仓库里，`Launch UI` 是最接近你想让 `Origin UI` 承担角色的本地现成资源。

结论：

- **当前本地展示层优先参考 `Launch UI`。**
- 如果后续再引入 `Origin UI`，也必须限制在页面块层，不得替代 `shadcn/ui` 作为基础控件来源。

## 4. 组件分层决策

### 4.1 基础 UI 底座

固定使用：

- `shadcn/ui`
- `Radix UI`
- `lucide-react`
- `next-themes`

说明：

- `Radix UI` 是交互原语层
- `shadcn/ui` 是基础控件层
- 业务项目不应直接手写弹窗、Popover、Tabs、Tooltip、Dropdown 的底层行为

### 4.2 学生端页面块

当前本地优先使用：

- `Launch UI`

后续可选引入：

- `Origin UI`
- `Magic UI`
- `Motion`

使用边界：

- `Launch UI / Origin UI` 用于 Hero、Stats、FAQ、CTA、Navbar、Footer、展示型 Cards
- `Magic UI` 只允许进入首页和宣传区
- `Motion` 只做轻量入场、切换、Hover、弹窗过渡

禁止：

- 把 `Magic UI` 用进后台、报名表、审核页、数据表格页
- 为了页面“显得高级”而给管理后台堆动画

### 4.3 后台业务层

固定使用：

- `@tanstack/react-table`
- `react-hook-form`
- `zod`
- `@hookform/resolvers`
- `sonner`

按需使用：

- `react-dropzone`
- `dnd-kit`
- `FullCalendar`

后置：

- `Plate`

## 5. 主模板当前已经自带的能力

`shadcn-dashboard-landing-template-main/nextjs-version` 当前已自带：

- `@tanstack/react-table`
- `react-hook-form`
- `@hookform/resolvers`
- `zod`
- `recharts`
- `sonner`
- `lucide-react`
- `next-themes`
- `@dnd-kit/core`
- `@dnd-kit/sortable`
- `@dnd-kit/utilities`

说明：

- 因此第一阶段不应机械重复安装这些依赖。
- 后续真正开工时，应先审计主模板已有依赖，再决定补装。

`launch-ui-main` 当前明显已带：

- `lucide-react`
- `next-themes`

但它不承担后台业务组件职责。

## 6. 当前缺失但可规划的组件库

以下能力当前仓库没有本地源码副本：

- `Origin UI`
- `Magic UI`
- `Tremor`
- `Plate`
- `react-dropzone`

对它们的项目口径应是：

- 这是**可选引入的后续依赖**
- 不是当前仓库已经具备的“本地组件源”

其中建议优先级：

### 可以较早引入

- `motion`
- `react-dropzone`

### 第二阶段再引入

- `@fullcalendar/react`
- `@fullcalendar/core`
- `@fullcalendar/daygrid`
- `@fullcalendar/interaction`

### 后置

- `Plate`
- `Tremor`
- `Origin UI`
- `Magic UI`

说明：

- `Tremor` 不是当前第一阶段必需品，因为后台首页先用 `shadcn/ui Card + Recharts` 就足够。
- `Plate` 也不是第一阶段必需品，比赛介绍和通知内容前期可以先用普通表单字段或 mock HTML。

## 7. 第一阶段依赖策略

第一阶段目标是做纯前端 Mock 与关键业务组件，不是把所有库一次装满。

建议顺序：

### 7.1 先复用主模板已有能力

- 表格：直接复用 `TanStack Table`
- 表单：直接复用 `React Hook Form + Zod + shadcn/ui Form`
- 图表：直接复用 `Recharts`
- 提示：直接复用 `Sonner`
- 图标：直接复用 `Lucide React`
- 主题：直接复用 `next-themes`
- 拖拽：后续若做字段排序，先用模板现有 `dnd-kit` 依赖

### 7.2 第一阶段真正可能补装的

- `motion`
- `react-dropzone`

### 7.3 第二阶段再补

- `@fullcalendar/react`
- `@fullcalendar/core`
- `@fullcalendar/daygrid`
- `@fullcalendar/interaction`

### 7.4 更后面再决定

- `Plate`
- `Tremor`
- `Origin UI`
- `Magic UI`

## 10. 最终口径

当前最稳的组件策略应理解为：

```text
UI 底座：
shadcn/ui
Radix UI
Lucide React
next-themes

学生端展示：
优先 Launch UI
Origin UI 为后续可选
Magic UI 少量使用
Motion 少量使用

后台业务：
TanStack Table
React Hook Form
Zod
Sonner

内容与材料：
react-dropzone
Plate 后置

统计与日程：
Recharts
FullCalendar 后置

高级交互：
dnd-kit
Command
```

要点只有两条：

- **先统一，再变美。**
- **先做 Mock 业务组件，再补重型内容编辑与视觉特效。**
