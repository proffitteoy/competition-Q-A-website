# 首页门面化改版方案（对标“学院竞赛门户”形态）

## 1. 背景与问题

当前首页虽然已经具备比赛与平台信息展示能力，但整体观感更接近“工作台入口页”，不足以承担学院官网级门户首页的门面职责，主要问题包括：

1. 首页叙事偏“系统骨架说明”，对访客的第一感知不够直观。
2. 门户导向信息密度不足，缺少“通知、重点赛事、资料入口”的主次分层。
3. 局部文案仍偏工程描述，不够贴近学生与教师的使用场景。

本次改版目标是把首页升级为“学院竞赛门户首页”，而不是营销站，也不是后台工作台。

## 2. 目标与非目标

## 2.1 目标

1. 首屏 3 秒内让用户明确：这是学院竞赛官方门户，能快速找到比赛并报名。
2. 强化首页三大核心区：`重点赛事`、`最新通知`、`资料下载/常见问题`。
3. 保持与现有 MVP 业务优先级一致：`报名与审核` > `竞赛门户` > `问答沉淀`。
4. 兼容移动端，保证首页在手机端可快速完成“找比赛-看通知-去报名”。

## 2.2 非目标

1. 不在本轮引入新的后台功能。
2. 不在本轮扩展完整社区化问答能力。
3. 不进行跨目录大规模重构，不改动无关路由与页面。

## 3. 对标形态拆解（参考 https://www.tipdm.org/）

说明：仅参考其“学院门户结构感与信息组织方式”，不复制对方品牌视觉、文案和具体布局。

可借鉴的门户特征：

1. 学院/平台身份明确，首页更像“官方入口”而非工具后台。
2. 导航层级清晰，首屏即提供核心办事入口。
3. 以“栏目化信息块”组织内容：赛事、公告、资料、常见问题。
4. 重视时效信息（近期赛事与公告）并给出明确跳转。

借鉴与不借鉴的落位说明：

| 参考特征 | 本项目借鉴方式 | 不借鉴内容 |
|---|---|---|
| 官方门户身份 | 落到导航、首屏标题、页脚机构信息 | 不复制对方品牌名、校名、标语 |
| 栏目化信息布局 | 落到“重点赛事 / 最新通知 / 资料与 FAQ”三大区块 | 不照搬对方页面结构与视觉层级 |
| 时效性内容前置 | 落到首页通知区与重点赛事区 | 不做资讯站式的大量长列表铺陈 |
| 办事入口明确 | 落到“查看比赛 / 报名指南 / 我的报名”入口 | 不把首页做成营销页或后台看板 |

## 4. 首页信息架构（V2）

建议首页模块顺序（从上到下）：

1. 顶部全局导航
- Logo/平台名、比赛列表、通知、我的报名、登录。
- 保留“立即查看比赛”主 CTA。

2. 门户首屏（Hero）
- 标题聚焦“学院竞赛官方门户”。
- 副标题简洁说明平台职责：发布、报名、审核、资料沉淀。
- 主按钮：`查看全部比赛`。
- 次按钮：`查看报名指南`（跳至资料区或比赛列表筛选）。

3. 快速入口条
- 报名中比赛
- 最新通知
- 资料下载
- 我的报名查询
- 阶段 A 先采用静态入口卡片，不强依赖实时计数。
- 阶段 B 再视情况补充计数或最近更新时间。

4. 重点赛事区（首页核心）
- 展示报名中/即将开始/进行中赛事。
- 每个卡片突出：比赛名、时间、状态、报名入口。
- 支持“查看更多赛事”跳转。

5. 最新通知区（首页核心）
- 展示最近公告列表（按发布时间倒序）。
- 每条至少包含：标题、所属比赛、时间、状态。
- 提供“查看全部通知”。
- 阶段 A 使用静态占位数据或与真实字段对齐的 mock 结构，不新增公开通知 API。
- 阶段 B 优先新增面向前台的只读读取能力，不复用 `/api/admin/notices`。
- 倾向实现为：首页服务端直接调用公开的 notice repository / service；只有在后续前台通知页需要客户端读取或筛选时，再补 `/api/notices`。

6. 资料与 FAQ 区（首页核心）
- 左侧：资料下载（报名模板、说明文档、常用附件）。
- 右侧：常见问题（报名资格、材料要求、审核时效等）。
- 阶段 A 可先使用静态配置数据，字段命名与未来正式数据结构保持一致。
- 阶段 B 倾向落到数据库实体，而不是长期停留在 JSON 配置文件。
- 推荐方向：新增“资料/附件”实体，与比赛建立关联，后台后续由 FAQ / 附件管理页统一维护。

7. 页脚
- 学院竞赛办公室联系方式、办公时间、版权信息。

## 4.1 现有首页模块去留决策

针对当前首页已有模块，先明确处理策略，避免实施歧义：

1. `PortalHero`
- 保留，但整体改写为“门户首屏”，去掉“统一工作台”“骨架”等工程叙事。

2. `PortalStats`
- 阶段 A 从首页主流程移除，不再作为独立区块保留。
- 原因是当前依赖 `@/lib/mock-data` 的 `platformStats`，且统计信息不是门户首页当前最高优先级。
- 阶段 B 若有真实且稳定的指标，再决定是否以“平台服务概览”形式低优先级回归。

3. `PortalFeatureGrid`
- 不保留当前版本，因为其内容和标题明显偏开发视角。
- 阶段 A 可将其替换为“快速入口条”或并入首屏下方的服务入口区。

4. `src/app/page.tsx` 中的独立赛事展示 `Section`
- 当前该区块以“当前门户骨架优先展示能报名、即将开始和正在进行中的比赛”为标题，属于页面内联结构，不是 `PortalHero` 自带区块。
- 阶段 A 不保留这一独立区块的现有形态，而是将其内容吸收进 V2 架构中的“重点赛事区”。
- 实施时应避免“Hero 右侧赛事 + 中段独立赛事 Section + 新重点赛事区”三处重复展示并存。

5. `PortalFaq`
- 保留，但文案改写为学生/教师视角的常见问题，而非“平台骨架边界”说明。

6. `PortalFooter`
- 保留，统一中文化并强化学院官方联系信息。

## 5. 视觉与文案方向

基于 `docs/视觉与交互规范.md`，首页视觉策略如下：

1. 风格关键词：`学院官方感`、`竞赛服务感`、`清晰可信`。
2. 视觉层次：首屏与核心栏目可增强，但不做过度营销动效。
3. 状态颜色严格复用语义色（如 `registration_open`、`upcoming` 等）。
4. 文案从“工程表述”改为“用户任务表述”，例如：
- “查看当前可报名比赛”
- “下载报名材料”
- “查看审核进度”
5. 所有面向用户的首页文案统一改为中文，不保留英文栏目标签或英文辅助标题。

阶段 A 需要清理的现有英文残留包括但不限于：

1. `Competition Hub`
2. `Featured Flow`
3. `What The Skeleton Fixes`
4. `College Competition Platform`

## 6. 组件与目录落位建议（仅规划，不落代码）

按现有分层，优先在以下目录演进：

1. `src/app/page.tsx`
- 负责首页模块编排顺序调整。

2. `src/components/marketing`
- 门户导航、首屏、栏目区块、页脚等展示组件。

3. `src/components/competitions`
- 复用现有比赛卡片，必要时增加“门户版紧凑卡片变体”。

4. `src/components/shared`
- 通用“通知列表项”“下载列表项”等跨页面复用块。

5. 数据读取策略
- 阶段 A：允许首页继续使用现有比赛数据读取方式，并为通知/资料区补充静态占位数据。
- 阶段 B：将首页数据拆分为“比赛、通知、资料”三类公开读取来源。
- 不建议让前台首页直接调用后台管理 API。

## 7. 分阶段实施计划

## 阶段 A：结构改版（最小可验证）

1. 调整首页模块顺序为“首屏-重点赛事-通知-资料/FAQ”。
2. 收敛文案语气为门户化表达。
3. 比赛区继续使用现有真实数据来源；通知区和资料区先用静态占位结构。
4. 移除或替换当前 `PortalStats`、`PortalFeatureGrid` 中偏工程叙事的内容。
5. 不处理导航栏登录态切换，导航仍保留静态“登录”入口，此项后置到阶段 B 或后续。

验收标准：

1. Hero 标题与副标题中不再出现“工作台”“骨架”“Skeleton”等工程术语。
2. 首屏或一屏内至少可见：赛事入口、通知/公告入口、主 CTA 按钮。
3. 当前首页所有面向用户的英文栏目标题与辅助标签全部清理为中文。
4. 移动端首屏 CTA 按钮可见、可点击，且不被导航或首屏图片遮挡。
5. 首页主结构不再依赖 `PortalStats` 作为信息主轴。

## 阶段 B：信息增强（同目录增量）

1. 为首页通知区接入公开只读的数据来源，不复用后台管理 API。
2. 资料下载区与 FAQ 区接入正式数据模型或后台可维护来源。
3. 视需要补充导航栏登录态展示，如用户名、个人入口、退出操作。
4. 补齐移动端栅格与间距细节，并清理首页对 `mock-data` 的直接依赖。

验收标准：

1. 首页三大核心区都由正式字段驱动，而不是长期依赖静态占位。
2. 首页不再直接引用 `@/lib/mock-data` 为门户区块提供数据。
3. 通知区、资料区的数据来源可被替换或维护，不依赖硬编码文案。
4. 移动端无关键信息折叠丢失，CTA 可直接点击。

## 8. 风险与约束

1. 若通知与资料数据接口尚未稳定，阶段 A 必须显式采用静态占位，不应临时复用后台管理接口。
2. 首页改版需避免引入后台风格组件，保持“前台门户”和“后台工作台”视觉角色分离。
3. 为控制范围，本轮不改动 `/admin`、报名表单流程与审核表格行为。
4. 导航栏登录态若纳入阶段 A，会扩大范围并引入会话判断与退出交互，默认不在本轮首批实现内。
