# next_discussion_platform 模板评估与吸收方案

## 1. 文档目的

本文用于评估根目录新增模板 `next_discussion_platform-main` 对当前项目的可用价值，并给出可执行的吸收策略。

本文结论只针对当前仓库阶段生效：

- 当前项目主定位仍是“学院竞赛管理与问答平台”
- 当前主链路仍是“竞赛门户 + 报名 + 审核 + 导出”
- 问答能力属于围绕竞赛的信息沉淀模块，不是独立论坛产品

## 2. 当前结论

`next_discussion_platform-main` 不应直接并入当前主项目，也不应作为新的主骨架继续开发。

当前建议是：

1. 将该目录视为“论坛能力参考模板”，而不是第二套主应用
2. 当前阶段只做结构评估与能力映射，不做整仓合并
3. 后续如果进入问答模块建设，只允许按需抽取思路和局部交互，再用当前项目技术栈重写

## 3. 模板现状判断

模板实际是完整论坛应用，而不是轻量展示组件库或局部页面模板。

已观察到的事实包括：

- 模板目录为 `next_discussion_platform-main/next_discussion_platform-main`
- 首页是帖子流和社区推荐结构，不是竞赛门户结构
- 内置社区、帖子、评论、投票、收藏、搜索、推荐等论坛能力
- 评论模型支持 `parentId` + `depth` 字段，是完整的树形嵌套回复结构，不是扁平评论列表
- 依赖 Firebase Auth、Firestore、Storage、Cloud Functions
- 基础 UI 使用 Chakra UI 与 Emotion
- 全局状态使用 Jotai

这意味着它和当前项目的定位、信息架构、数据层路线都不一致。

## 4. 与当前项目路线的冲突点

### 4.1 产品定位冲突

当前项目不是泛论坛，也不是社区优先产品。

当前平台首页、比赛详情页、报名页、审核台都要求围绕“某个比赛”的业务主线组织信息。论坛模板则默认围绕：

- 社区
- 帖子流
- 订阅关系
- 投票热度
- 个人化 feed

如果直接引入，会把项目中心从“比赛服务”带偏成“社区讨论”。

### 4.2 技术栈冲突

当前项目默认路线已经固定为：

- Next.js
- TypeScript
- Tailwind CSS
- shadcn/ui
- PostgreSQL
- Drizzle
- Auth.js

论坛模板则明显走另一条路线：

- Chakra UI
- Emotion
- Firebase
- Cloud Functions
- Jotai

如果直接并入，会同时引入两套：

- UI 体系
- 状态管理方式
- 认证与数据存储方式
- 项目目录组织方式

这会显著抬高维护成本，也会削弱当前仓库已经形成的组件分层约束。

需要注意的是，两套项目并非完全不兼容。以下技术选型存在重叠：

- 表单验证：两者都使用 `react-hook-form` + `zod`，验证 schema 的设计思路可以直接复用
- 主题切换：模板使用 `next-themes`，当前项目也有 `theme-provider.tsx`，不构成冲突
- 测试工具链：模板配置了 `vitest` + `@testing-library/react`，当前项目可参考其测试组织方式

冲突集中在 UI 体系、状态管理、认证与数据层，表单验证和测试工具链层面是互通的。

### 4.3 阶段优先级冲突

当前项目文档已经明确：

- MVP 优先完成竞赛门户、报名、审核、导出
- 问答属于后续阶段
- 不优先投入大而全的论坛化首页、复杂运营玩法和社区体系

论坛模板最强的部分恰好是：

- 社区运营
- 帖子流
- 互动反馈
- 收藏和热度

这些内容现在不应反向主导项目排期。

## 5. 当前可接受的使用边界

当前仓库对这个模板的使用边界应明确如下：

1. 不直接运行成第二套业务主站
2. 不把 `next_discussion_platform-main` 当作当前项目源码目录继续开发
3. 不直接迁入 Firebase、Firestore、Cloud Functions 作为当前项目正式后端
4. 不直接引入 Chakra UI 作为当前项目基础组件体系
5. 不把“社区”“订阅”“帖子流推荐”当作当前项目首页主结构

如果后续需要借鉴，只允许：

1. 借鉴问答型页面的信息结构
2. 借鉴帖子详情页的评论展开方式
3. 借鉴搜索、推荐、互动入口的产品思路
4. 将可用交互重写到当前的 `src/` 主项目结构内

## 6. 可吸收能力映射

论坛模板不是完全无用，但只能按“未来问答模块参考源”来吸收。

### 6.1 可参考能力与优先级

可优先研究并转译的能力包括，但不应一次性同时落地。

当前建议按“问答 MVP 必须能力”与“增强能力”分层：

| 模板能力 | 当前项目未来可映射方向 | 优先级 | 说明 |
|:---|:---|:---|:---|
| 帖子列表 | 问题列表 | P0 | 问答模块 MVP 必须能力 |
| 帖子详情 | 问题详情 | P0 | 问答模块 MVP 必须能力 |
| 评论与楼中楼回复 | 问答评论与补充讨论 | P0 | 用于承接追问、补充说明和澄清 |
| 互动状态反馈 | 问答交互反馈 | P1 | 指加载、提交、成功失败反馈，不等于复杂运营玩法 |
| 搜索弹窗 | 竞赛问答检索 | P1 | 适合在问答内容规模起来后接入 |
| 推荐区块 | 相关问题推荐 | P1 | 适合放在问题详情页，不应先做首页内容流 |
| 收藏能力 | 问题收藏 | P2 | 属于增强能力，不进入问答 MVP 首批范围 |
| 嵌套评论数据模型（`parentId` + `depth`） | 问答评论表设计 | P0 | 模板的 Comment 类型支持树形回复，是问答评论表最直接的参考模型 |
| 业务 hooks 分层模式 | 问答模块 hooks 组织 | 工程参考 | 模板按领域分目录（`hooks/posts/`、`hooks/comments/`），每个操作独立 hook（如 `usePostVote`、`useCreateComment`），当前项目 `src/hooks/` 只有 UI 工具类 hook，后续问答模块应参考此分层方式 |
| 级联清理模式 | 问答内容删除策略 | 工程参考 | 模板通过 Cloud Functions 触发器实现"帖子删除时级联清理收藏记录"，虽然当前项目不用 Firebase Functions，但级联删除的业务逻辑（问题删除时清理回答、评论、收藏）是通用需求，应在问答数据模型设计时预先考虑 |

优先级解释：

- `P0`：问答 MVP 必须能力，进入问答模块时优先实现
- `P1`：问答模块增强能力，在 MVP 可用后补充
- `P2`：可延后能力，不应阻塞问答模块首版落地

### 6.2 不建议直接映射的能力

以下能力与当前产品边界不匹配，不建议直接继承：

| 模板能力 | 不建议直接迁入的原因 |
|:---|:---|
| 社区创建与订阅 | 当前项目的核心对象是比赛，不是社区 |
| 首页个性化 feed | 当前首页应优先做比赛分发，不是内容流 |
| 基于投票的公共内容排序 | 当前问答优先级低于报名主链路，且需结合竞赛语境做约束 |
| 独立社区后台 | 容易形成第二套管理模型，和比赛后台冲突 |
| 基于 Firebase 的用户资料同步 | 与现有用户、认证、数据库路线冲突 |

## 7. 推荐吸收路径

### 7.1 阶段 A：当前阶段只做调研，不做并入

当前阶段目标：

- 保留模板目录作为参考源
- 不把模板代码直接搬入业务主链路
- 在文档层明确吸收边界

当前阶段交付物：

- 本评估文档
- 后续如有需要，再补“问答模块原型方案”

### 7.2 阶段 B：进入问答模块时做结构抽取

触发条件：

- 报名、审核、导出主链路已完成真实数据验证，而不是只停留在 mock 演示
- 比赛详情页已稳定承接 FAQ、附件、通知，不再频繁改动信息结构
- 问答模块已正式进入排期

建议把“可以进入阶段 B”细化为以下可判定门槛：

1. 至少 1 个真实比赛已完成“发布 -> 报名 -> 审核 -> 导出”全链路验证
2. 管理员已在真实数据环境完成基础审核与导出操作，不依赖临时脚本兜底
3. 学生侧“我的报名”与管理员侧“报名审核”已能稳定读取真实数据
4. 比赛详情页中的 FAQ、附件、通知已形成稳定分区，不再处于频繁改版期
5. 团队确认问答模块不会反向阻塞报名主链路迭代

届时建议先做：

1. 只抽取页面结构和交互模式
2. 首批只保留与”问题、回答、评论”直接相关的部分，收藏和推荐留到问答 MVP 可用后的增强阶段（对应 6.1 节 P1/P2 优先级）
3. 在当前项目目录中重写（具体路由归属见下文）：
   - `src/app/competitions/[id]/questions`
   - `src/app/competitions/[id]/questions/[qid]`
   - `src/components/questions`
   - `src/server/repositories/question-repository.ts`
4. 数据层统一走当前项目正式路线，不保留 Firebase 依赖

关于路由归属的初步方向：

由于当前阶段问题必须绑定比赛（见 7.3 节），路由设计应体现这一从属关系：

- 学生侧页面：`src/app/competitions/[id]/questions` 和 `src/app/competitions/[id]/questions/[qid]`，问答作为比赛详情的子页面
- 数据操作：查询走 repository，写操作优先走 Server Actions，与现有报名链路保持一致；仅在确有需要（如第三方接入、文件下载）时补 Route Handler，不默认为问答新建一套 API CRUD 入口
- 管理后台：嵌入现有的 `src/app/(dashboard)/admin/competitions` 管理流程，不单独建 `admin/questions` 顶级入口
- 组件目录：`src/components/questions/`，与现有 `competitions/`、`admin/` 平级
- 数据层：`src/server/repositories/question-repository.ts`，与现有 repository 平级

如果后续允许不绑定比赛的通用问题，再考虑提升为顶级路由 `src/app/questions`。当前不预埋。

### 7.3 阶段 C：与比赛详情页做业务绑定

问答模块落地时，应以比赛为主键组织，而不是以社区为主键组织。

推荐关系应为：

- 问答 MVP 阶段只允许“绑定比赛的问题”，不引入未绑定比赛的通用问题
- 一个问题绑定一个比赛，作为当前阶段默认模型
- 比赛详情页显示该比赛相关问题
- FAQ 与问答互为补充
- 管理员能对问题进行审核、隐藏、置顶、精选

关于“是否允许不绑定比赛的通用问题”，当前立场是：

- 当前不允许
- 如果后续确有跨比赛公共问答需求，应作为单独扩展决策重新评审
- 在那之前，不为“通用问题”预埋第二套首页结构、路由语义或数据模型分支

不建议出现：

- 独立于比赛的社区空间
- 论坛化首页替代竞赛门户首页
- 以社区运营替代比赛通知与资料体系

## 8. 对实现的直接约束

如果后续开始吸收该模板，实施时必须遵守以下规则：

1. 只在当前主项目目录下开发，不在模板目录里继续叠业务代码
2. 只复制必要代码，不整目录照搬
3. UI 统一收敛到 `shadcn/ui + Tailwind CSS`
4. 状态管理优先复用当前项目已有模式，不额外引入 Jotai 作为新默认
5. 认证与数据统一走当前项目数据层，不引入 Firebase 作为正式依赖
6. 页面命名、路由结构、实体命名必须围绕“比赛问答”，不能照搬“社区帖子”语义

### 8.1 模板目录与 Git 管理策略

对于根目录下的 `next_discussion_platform-main`，当前建议采用以下策略：

1. 当前评估阶段不把它并入主项目源码，也不让它进入正式业务目录
2. 如果只是本地调研使用，则不纳入版本管理，避免把整套论坛应用误当成当前项目一部分
3. 如果团队后续确实需要共享这份参考模板，则不应继续放在仓库根目录，而应单独迁移到 `references/next_discussion_platform-main`
4. 迁移到 `references/` 时，应同时补一份来源说明，写清上游仓库、版本、用途和禁止直接开发的约束
5. 当问答模块完成吸收、且模板已不再提供额外参考价值后，应删除该模板目录，避免长期制造结构噪音

换句话说，当前不建议把它长期以 `untracked` 的形式留在根目录，也不建议把它直接当成正式项目目录提交。

## 9. 建议的后续文档

如果确认后续要继续推进问答模块，建议在本方案之后补 1 份合并文档：

1. `docs/竞赛问答模块方案.md`

该文档应统一回答：

- 问题、回答、评论、收藏、标签的最小数据模型是什么
- 问题如何绑定比赛
- FAQ 和问答如何联动
- 什么内容可以公开，什么内容需要审核
- 后台如何管理问答内容
- 页面结构与路由如何设计
- 哪些能力属于 MVP，哪些能力延后

## 10. 本轮最终建议

本轮对 `next_discussion_platform-main` 的处理建议如下：

1. 保留目录，作为后续问答模块的参考模板
2. 不把它作为当前项目主骨架
3. 不在当前阶段接入其 Firebase、Chakra UI、Jotai 方案
4. 等主链路稳定后，再按“竞赛问答”语义抽取并重写局部能力

换句话说，这个模板当前的角色不是“接入模板”，而是“后续问答能力参考源”。
