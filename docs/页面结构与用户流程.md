# 页面结构与用户流程

## 1. 页面清单

### 学生端 / 公共端

| 页面 | 目标 | 主要内容 | 主要动作 |
|:---|:---|:---|:---|
| 首页 | 快速分发当前重点比赛与通知 | Banner、报名中比赛、最新通知、FAQ/问答入口、资料下载入口 | 浏览、搜索、进入比赛详情 |
| 比赛列表 | 按状态筛选比赛 | 状态筛选、搜索、比赛卡片、时间与状态标记 | 筛选、查看详情 |
| 比赛详情 | 承接某个比赛全部核心信息 | 比赛介绍、时间安排、报名状态、通知、FAQ、附件、相关问答 | 报名、下载、查看通知、提问 |
| 登录页 | 完成身份校验 | 账号登录、找回密码入口、统一认证预留 | 登录 |
| 报名页 | 提交报名信息 | 动态报名表、个人/团队模式、校验提示、提交确认 | 填写、暂存、提交 |
| 我的报名 | 查看报名历史和审核结果 | 我的报名记录、状态、审核意见、导出回执 | 查看详情、补充信息 |
| 问答列表 | 浏览竞赛相关问题 | 问题列表、标签、比赛筛选 | 查看、提问 |
| 问答详情 | 查看问题沉淀 | 问题正文、回答、评论、采纳状态 | 回答、评论、收藏 |

### 管理后台

| 页面 | 目标 | 主要内容 | 主要动作 |
|:---|:---|:---|:---|
| 比赛管理 | 创建与维护比赛 | 列表、搜索、状态、负责人、时间设置 | 新建、编辑、发布、下线 |
| 报名表管理 | 配置比赛报名字段 | 字段列表、字段类型、必填、校验规则 | 新增字段、排序、启停 |
| 报名审核 | 承接核心业务处理 | 报名表格、筛选器、详情抽屉、审核日志 | 通过、驳回、撤回、批量处理、导出 |
| 通知管理 | 维护比赛通知 | 绑定比赛、状态、发布时间、排序 | 发布、撤回、排序 |
| 赛程管理（占位） | 保留后台赛程路由与后续日历接入点 | 简版时间线、关键日期维护、后续日历说明 | 查看、维护关键时间节点 |
| FAQ / 附件管理 | 维护资料沉淀 | FAQ 列表、附件列表、归属比赛 | 新增、编辑、删除 |
| 用户与权限 | 管理账号和角色 | 用户列表、状态、角色、作用域 | 设角色、禁用、分配比赛 |
| 问答管理 | 管控内容质量 | 问题/回答列表、审核状态、置顶精华 | 审核、隐藏、置顶 |

## 2. MVP 页面范围

第一版只固定以下页面：

- 首页
- 比赛列表
- 比赛详情
- 登录页
- 报名页
- 我的报名
- 管理后台的比赛管理
- 管理后台的报名审核
- 数据导出入口

问答管理页、FAQ 管理页、附件管理页可先以占位结构保留，不必一开始做完整交互。
`/admin/schedule` 路由同样保留，但第一版只要求占位页或简版时间线，不要求完整日历交互。

## 3. 每页信息架构

### 3.1 首页

推荐模块顺序：

1. 顶部导航：比赛、通知、问答、我的报名、登录
2. 头图区域：主标题、平台定位、当前报名中的重点比赛 CTA
3. 报名中比赛区：按优先级展示 3 到 6 个比赛卡片
4. 最新通知区：按比赛归属展示最新通知
5. 资料下载入口：模板、指南、常见附件
6. 热门问答区：减少重复咨询
7. 页脚：帮助、联系方式、版权

### 3.2 比赛列表

必须具备：

- 状态筛选：报名中、即将开始、进行中、已结束
- 搜索框：按比赛名称、关键词搜索
- 列表视图：封面、名称、时间、状态、报名按钮
- 空状态：没有匹配比赛时给出清晰提示

### 3.3 比赛详情

推荐分区：

1. 首屏摘要：名称、状态、报名时间、负责人、报名 CTA
2. 比赛介绍：参赛对象、赛道、规则说明
3. 时间安排：报名、初赛、复赛、结果公布
4. 通知公告：只显示当前比赛相关通知
5. FAQ：当前比赛常见问题
6. 附件下载：报名模板、说明书、素材包
7. 相关问答：围绕该比赛的问题沉淀

### 3.4 报名页

必须解决：

- 区分个人报名与团队报名
- 对动态表单字段进行前端校验
- 明确重复报名限制
- 在提交前展示确认摘要
- 提交后反馈状态和后续步骤

### 3.5 报名审核页

必须解决：

- 按比赛、学院、专业、年级、状态筛选
- 列表和详情联动
- 直接查看报名原始信息和审核日志
- 单条和批量审核
- 一键导出当前筛选结果

## 4. 用户操作路径

### 4.1 学生主路径

1. 首页或比赛列表发现比赛
2. 进入比赛详情阅读规则与附件
3. 点击报名并登录
4. 填写报名表并提交
5. 在“我的报名”查看审核状态
6. 如被驳回，根据意见修改后再次提交或补充材料

### 4.2 管理员主路径

1. 创建比赛并填写基础信息
2. 配置报名表字段与报名时间
3. 发布比赛与相关通知
4. 在审核页按条件筛选报名记录
5. 查看详情并执行通过/驳回/撤回
6. 导出报名数据给线下评审或学院存档

## 5. 关键状态设计

### 5.1 比赛状态

- `draft`：草稿
- `upcoming`：即将开始
- `registration_open`：报名中
- `in_progress`：进行中
- `finished`：已结束
- `archived`：归档

说明：

- 前台列表的“状态筛选”应服务用户认知，后台内部状态可更细，但不要让展示文案和业务状态脱节。
- 合法转换与触发条件以 [状态流转矩阵](./状态流转矩阵.md) 为准。

### 5.2 报名状态

- `draft`：暂存未提交
- `submitted`：已提交待审核
- `approved`：审核通过
- `rejected`：审核驳回
- `withdrawn`：已撤回
- `cancelled`：管理员取消

说明：

- 每次人工审核动作都必须写入审核日志。
- 如果允许驳回后补交，需要明确是“修改原记录再提交”还是“生成新版本”。
- 当前项目默认采用“同一主记录重提 + revision 快照”方案，详见 [报名表数据模型与导出约束](./报名表数据模型与导出约束.md) 与 [状态流转矩阵](./状态流转矩阵.md)。

### 5.3 用户状态

- `active`：正常
- `pending_verification`：待验证
- `disabled`：禁用

### 5.4 问答状态

主状态：

- `pending_review`：待审核
- `published`：已发布
- `hidden`：已隐藏

附加标记：

- `is_featured`：是否精华
- `is_pinned`：是否置顶

说明：

- `is_featured` 与 `is_pinned` 不是互斥状态，不应与主状态共用同一枚举字段。

## 6. 管理员工作流

### 6.1 比赛发布工作流

1. 创建比赛草稿
2. 配置基础信息与时间
3. 绑定通知、FAQ、附件
4. 配置报名表
5. 检查前台展示效果
6. 切换为可见或报名中

### 6.2 报名审核工作流

1. 进入指定比赛的报名审核页
2. 按状态和学院等条件筛选
3. 打开记录详情查看报名信息
4. 填写审核意见
5. 执行通过、驳回、撤回等动作
6. 自动写入审核日志
7. 需要汇总时导出筛选结果

## 7. 实现提醒

- 比赛详情页必须成为该比赛的“单一信息入口”，避免通知、FAQ、附件、问答彼此割裂。
- 报名审核页应优先做成强业务表格页，不要先追求装饰性统计看板。
- 问答页后续可扩展，但第一版不应干扰报名主链路。
