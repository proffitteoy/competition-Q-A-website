# 数据层与后端架构

## 1. 目标与结论

本文件用于补齐阶段 2 “接真数据”前必须冻结的后端决策，避免前端 Mock 通过后因数据层方案反复返工。

当前默认结论如下：

- 应用形态：`Next.js App Router` 单体应用，前后端同仓
- 数据库：`PostgreSQL`
- ORM / Migration：`Drizzle ORM + drizzle-kit`
- 鉴权：`Auth.js` 数据库 Session 方案
- 读写入口：`Server Actions + Route Handlers` 组合
- 文件存储：`S3 兼容对象存储`
- 部署方式：`Docker 化单体 Next.js + PostgreSQL + 对象存储`

说明：

- 以上是阶段 2 的默认真相源，不再继续把后端能力悬空到“实现时再看”。
- 若后续要替换其中任一项，必须先说明对报名主链路、审核链路、导出链路的影响。

## 2. 架构边界

平台当前不是“前端站点 + 外置通用后端”拆分模式，而是围绕竞赛平台主链路做一体化交付：

```text
Next.js UI
-> Server Actions / Route Handlers
-> Service / Permission Layer
-> Drizzle ORM
-> PostgreSQL
-> S3 Compatible Object Storage
```

约束如下：

- 不引入独立 BFF、微服务或第二套管理 API。
- 不在阶段 2 引入 tRPC。
- 不把业务写死在 React 组件内部，权限、状态流转、重复报名校验必须沉到服务层。

## 3. 数据库选型

### 3.1 结论

- 生产和测试环境数据库统一使用 `PostgreSQL`
- 动态表单、报名快照、审核日志等半结构化数据统一落 `JSONB`

### 3.2 原因

- 竞赛、报名、审核、权限作用域本质是强关系型数据。
- 动态报名表、报名快照、导出映射又要求一定的半结构化能力。
- `PostgreSQL` 同时覆盖关系约束、`JSONB`、事务、一致性和后续统计扩展。

### 3.3 明确不选

- `SQLite`：不作为阶段 2 真实数据方案，只可用于极小范围本地实验。
- “先随便接一个云表 / 文档库”：会直接削弱状态流转、唯一约束和审核日志一致性。

## 4. ORM 与迁移策略

### 4.1 结论

- 统一使用 `Drizzle ORM + drizzle-kit`

### 4.2 采用原因

- 需要直接控制 `PostgreSQL` 的 `JSONB`、唯一约束、索引和枚举字段。
- 动态表单和状态日志对“数据库结构可控”要求高于“代码生成体验”。
- 单体 Next.js 项目中，`Drizzle` 足够轻，便于把 schema、migration、查询一起放在仓库内管理。

### 4.3 使用约束

- schema 文件统一放在 `src/lib/db/schema/*`
- 数据库客户端统一从 `src/lib/db/client.ts` 输出
- migration 文件统一纳入版本控制，不允许只靠本地临时变更
- 复杂查询可以写 SQL，但必须留在 repository / service 层，不直接散落页面

## 5. API 形态

### 5.1 结论

- 表单提交、后台审核、基础 CRUD 优先走 `Server Actions`
- 文件下载、导出、上传回调、第三方接入入口走 `Route Handlers`

### 5.2 选择原则

适合 `Server Actions` 的场景：

- 学生提交报名
- 学生撤回或补交
- 管理员单条审核
- 管理员创建 / 编辑比赛、通知、FAQ

适合 `Route Handlers` 的场景：

- `Excel / CSV` 导出
- 附件 / 录取名单等文件下载
- 对象存储上传签名
- 未来校园统一认证回调

### 5.3 约束

- 页面层不直接拼 SQL，也不直接绕过服务层写库。
- 无论入口是 `Action` 还是 `Route Handler`，都必须先过权限检查，再做业务校验。

## 6. 认证与会话

### 6.1 结论

- 统一使用 `Auth.js`
- 阶段 2 的真实登录采用 `Credentials` 或受控账号源
- Session 持久化到 `PostgreSQL`

### 6.2 角色模型

当前固定四类角色：

- `super_admin`
- `competition_admin`
- `content_editor`
- `student_user`

### 6.3 权限执行原则

- 认证解决“你是谁”
- 权限层解决“你能操作哪场比赛、哪些资源、哪些动作”
- `competition_admin` 的比赛作用域必须落到数据库，而不是只靠前端隐藏按钮

### 6.4 最小权限矩阵

| 角色 | 比赛 | 报名 | 通知 / FAQ / 附件 | 用户与角色 |
|:---|:---|:---|:---|:---|
| `super_admin` | 全部读写 | 全部审核 / 导出 | 全部读写 | 全部读写 |
| `competition_admin` | 仅自己负责的比赛读写 | 仅自己负责比赛的审核 / 导出 | 仅自己负责比赛的读写 | 不管理全站用户 |
| `content_editor` | 只读比赛上下文 | 不审核报名 | 可编辑被授权比赛的通知 / FAQ / 附件 | 不管理角色 |
| `student_user` | 公开信息只读 | 仅自己的报名读写 | 公开信息只读 | 仅维护自己的资料 |

说明：

- `content_editor` 默认不拥有全部比赛的内容编辑权，应支持按比赛作用域分配。
- 权限校验必须同时检查 `role` 与 `competition_id` 作用域。

### 6.5 `role_assignment` 建议结构

`role_assignment` 是权限作用域落库的核心表，阶段 2 至少应包含以下字段：

| 字段 | 含义 |
|:---|:---|
| `id` | 主键 |
| `user_id` | 被授予权限的用户 |
| `role` | 角色枚举 |
| `scope_type` | `global / competition` |
| `competition_id` | 比赛作用域；全站角色时可为空 |
| `granted_by` | 授权人 |
| `created_at` | 授权时间 |
| `revoked_at` | 撤销时间，可为空 |

规则：

- `super_admin` 默认使用 `scope_type = global`
- `competition_admin` 与 `content_editor` 默认使用 `scope_type = competition`
- 同一用户可持有多条作用域不同的授权记录
- 唯一约束至少应覆盖“同一用户 + 同一角色 + 同一作用域”仅有一条有效授权

## 7. 文件上传与存储

### 7.1 结论

- 不使用应用服务器本地磁盘作为长期文件存储
- 使用 `S3` 兼容对象存储

### 7.2 环境建议

- 本地开发：`MinIO`
- 生产环境：阿里云 `OSS`、腾讯云 `COS`、AWS `S3` 或其他兼容实现

### 7.3 约束

- 数据库只存文件元数据和对象 key，不存二进制大文件
- 上传前先做文件类型、大小、归属比赛与用途校验
- 竞赛附件、报名附件、导出文件分不同前缀目录管理

### 7.4 平台级默认上传约束

字段级 `validation_json` 可以收紧上传规则，但不能放宽平台级默认约束。

默认值建议如下：

| 项目 | 默认值 |
|:---|:---|
| 单文件最大体积 | `20MB` |
| 单次提交累计体积 | `100MB` |
| 默认允许类型 | `.pdf .doc .docx .xls .xlsx .ppt .pptx .zip .rar .7z .png .jpg .jpeg` |
| 默认禁止类型 | 可执行文件、脚本文件、未知二进制 |

说明：

- 比赛公共附件如需更大体积，可由服务端按用途单独放宽。
- 报名附件若未配置字段级规则，则继承上述默认值。

## 8. 部署目标

### 8.1 结论

阶段 2 默认部署目标为：

- `Docker` 化部署的 `Next.js` 单体应用
- 独立 `PostgreSQL`
- 独立对象存储服务或云对象存储

### 8.2 原因

- 学院项目通常需要更稳定的内网 / 校园网可达性和可控备份策略
- 比起把核心链路拆散到多平台，单体部署更利于早期交付和运维

### 8.3 不做的事

- 现阶段不为“未来可能很大规模”提前拆微服务
- 不先做多活、多租户、多区域架构

## 9. 推荐目录落位

```text
src/
  app/
    api/                     # 导出、上传签名、回调等 Route Handlers
  actions/                   # Server Actions
  lib/
    auth/                    # Auth.js 配置、session helper
    db/
      client.ts
      schema/
      migrations/
  server/
    permissions/             # 角色与比赛作用域校验
    repositories/            # 数据访问
    services/                # 业务规则、状态流转、重复报名校验
```

说明：

- `server/services` 是业务真规则层。
- 状态流转、重复报名校验、导出字段组装必须在这里集中处理。

## 10. 数据对象总览

阶段 2 必须优先落地以下对象：

- `user`
- `role_assignment`
- `competition`
- `competition_notice`
- `competition_attachment`
- `competition_faq`
- `competition_status_log`
- `registration_form`
- `registration_form_version`
- `registration_form_field`
- `registration`
- `registration_team_member`
- `registration_revision`
- `registration_audit_log`

### 10.1 实体关系总览

主要关系如下：

| 源实体 | 目标实体 | 关系 | 说明 |
|:---|:---|:---|:---|
| `user` | `role_assignment` | `1:N` | 一个用户可持有多条不同作用域授权 |
| `competition` | `role_assignment` | `1:N` | 比赛作用域授权绑定到具体比赛 |
| `competition` | `competition_notice / competition_attachment / competition_faq` | `1:N` | 比赛聚合通知、附件、FAQ |
| `competition` | `competition_status_log` | `1:N` | 比赛状态变化留痕 |
| `competition` | `registration_form` | `1:1` | 阶段 2 默认每场比赛一套逻辑报名表 |
| `registration_form` | `registration_form_version` | `1:N` | 报名表按版本演进 |
| `registration_form_version` | `registration_form_field` | `1:N` | 每个版本包含多字段定义 |
| `competition` | `registration` | `1:N` | 一场比赛有多条报名记录 |
| `user` | `registration` | `1:N` | 一个学生可参与多场比赛 |
| `registration_form_version` | `registration` | `1:N` | 报名记录绑定提交时表单版本 |
| `registration` | `registration_team_member / registration_revision / registration_audit_log` | `1:N` | 团队成员、修订快照、审核日志均从属于主报名记录 |

## 11. 阶段 2 落地顺序

### 11.1 审核能力的阶段边界

阶段 2 只要求落最小可用审核闭环：

- 管理员可通过受控入口执行单条审核动作
- 状态写入、权限检查、审核日志落库全部生效
- 不要求完整筛选工作台、批量处理和导出 UI

阶段 3 再补全：

- 审核工作台筛选
- 详情抽屉 / 批量处理
- 导出与统计筛选

建议按以下顺序推进：

1. 建 `user / role_assignment / competition` 基础表
2. 建 `competition_status_log` 与 `registration_form* / registration*` 表
3. 落 `Auth.js` 登录与 Session
4. 落学生报名提交、我的报名查询
5. 落最小可用管理员审核动作、审核日志
6. 落导出与上传

## 12. 本文解决的问题

本文明确了以下此前未冻结的事项：

- `PostgreSQL` 是正式数据库，不再悬而未决
- ORM 统一为 `Drizzle`
- API 形态是 `Server Actions + Route Handlers`
- 认证用 `Auth.js + DB Session`
- 文件存储使用 `S3` 兼容对象存储
- 部署按 `Docker 化单体` 推进
