# 技术选型与模板映射

## 1. 当前结论

本项目前端与全栈重写，优先采用：

- `Next.js`
- `TypeScript`
- `Tailwind CSS`
- `shadcn/ui`

组件层具体分工与依赖分期，详见 [组件选型与分层策略.md](./组件选型与分层策略.md)。

不优先采用：

- `Ant Design Pro`

原因不是它不能做，而是它默认的后台气质更接近传统企业后台，容易把学院竞赛平台做成“教务系统风格”，不符合本项目对门户质感和整体美观度的要求。

## 2. 本地模板调研结果

### 2.1 主模板：`shadcn-dashboard-landing-template-main`

本地目录特征：

- 同时包含 `nextjs-version` 和 `vite-version`
- `nextjs-version` 采用 App Router
- 已拆分为 `(auth)`、`(dashboard)`、`landing` 三类路由
- 已内置 Dashboard、Landing、Users、Tasks、Calendar、Mail、FAQ、Settings 等页面

对本项目的价值：

- 一套模板同时覆盖学生前台基础页和管理员后台
- 已经有较成熟的后台布局、表格、表单、导航和主题系统
- 适合承接比赛管理、报名审核、用户管理、数据表格、认证页

结论：

**`shadcn-dashboard-landing-template-main/nextjs-version` 应作为主实现骨架。**

### 2.2 辅助模板：`launch-ui-main`

本地目录特征：

- 是单独的 Next.js 站点
- `app` 结构很轻
- 组件集中在 `components/sections/*`
- 已有 `hero`、`faq`、`stats`、`cta`、`navbar`、`footer`、`items` 等门户展示组件

对本项目的价值：

- 适合加强学院竞赛门户首页的首屏、统计、特色区、FAQ、CTA、页脚
- 适合做“学院竞赛中心”的视觉表达，而不是后台业务页面

结论：

**`launch-ui-main` 更适合作为组件来源库，不适合单独承担整个平台主骨架。**

## 3. 模板使用策略

### 主干架构

- 主模板：`ShadcnStore Dashboard + Landing Template`
- 辅助模板：`Launch UI`

### 实施原则

- 后台、认证、业务表格、任务流以 `ShadcnStore` 为基础
- 学生端首页和营销化展示区，按需引入 `Launch UI` 的 section 组件设计语言
- 不维护两套并行主应用，避免后续路由、主题、组件体系分裂

## 4. 当前落地目录

当前代码骨架已经落到仓库根目录，实际项目主目录变为：

- `F:\竞赛问答平台\src`
- `F:\竞赛问答平台\public`

来源关系：

- 主骨架来源：`F:\竞赛问答平台\shadcn-dashboard-landing-template-main\nextjs-version`
- 首页展示参考：`F:\竞赛问答平台\launch-ui-main`

当前使用方式：

- 根项目承接 App Router、Dashboard Layout、Auth Layout、基础 UI 组件体系
- 公开端页面与后台业务页都应继续在根项目内演进
- 外部模板目录后续只作为补充参考或待删除来源，不再当作主项目目录开发

`launch-ui-main` 的建议使用方式：

- 只参考 `components/sections/*`
- 抽取可复用 section 结构与视觉风格
- 不建议把整个 `launch-ui-main` 作为第二套应用长期并存

## 5. 模板模块到业务模块的映射

### ShadcnStore 模块映射

| 模板原始模块 | 项目目标模块 |
|:---|:---|
| `landing` | 学院竞赛门户基础页 |
| `dashboard` / `dashboard-2` | 管理员后台首页 |
| `tasks` | 报名审核任务台 |
| `calendar` | 竞赛日程 |
| `mail` | 通知中心或站内消息 |
| `chat` | 问答/咨询辅助界面参考 |
| `users` | 用户管理 |
| `faqs` | FAQ 管理与展示参考 |
| 表格与设置页 | 比赛列表、报名列表、用户列表、系统配置 |

### Launch UI 组件映射

| Launch UI 组件 | 项目目标模块 |
|:---|:---|
| `hero` | 学院竞赛门户首页头图 |
| `stats` | 已举办比赛数、参赛人数、获奖数量 |
| `items` | 竞赛分类、平台能力说明、流程卡片 |
| `faq` | 报名常见问题 |
| `cta` | 立即查看比赛 / 登录报名 |
| `navbar` | 前台站点导航 |
| `footer` | 学院信息、联系方式、备案信息 |

## 6. 为什么不建议优先选 Ant Design Pro

- 它更容易导向传统企业后台视觉
- 首页和公开页往往还需要额外再做一层门户包装
- 本项目最大问题不是“缺后台功能”，而是“缺统一且好看的产品化前台与后台体验”
- 当前两个本地模板已经覆盖了本项目更需要的视觉与交互方向

## 7. 对后续开发的直接约束

- 第一版默认走 Next.js 路线，不再优先考虑 Vite 版本
- 公共端与后台端尽量共用一套设计令牌、颜色变量和基础组件
- 首页不要直接照搬后台模板 landing，也不要直接把 Launch UI 全站原样套上
- 应以竞赛业务信息架构为主，模板只提供骨架和组件，不提供产品结构

## 8. 当前已知风险

- 两个模板虽然同属 `Next.js + Tailwind + shadcn/ui` 路线，但版本并不完全一致。
- `shadcn-dashboard-landing-template-main/nextjs-version` 当前带 `pnpm-lock.yaml`，`launch-ui-main` 当前带 `package-lock.json`，包管理器口径不一致。
- 因此后续集成方式应是“以主模板为宿主，按需迁移 `Launch UI` section 组件”，而不是直接把两个项目整仓合并。
