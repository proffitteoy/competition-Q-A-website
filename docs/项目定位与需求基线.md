# 项目定位与需求基线

## 1. 项目一句话定义

**学院竞赛管理与问答平台**，用于承接学院竞赛从信息发布、报名、审核、导出到常见问题沉淀的完整业务闭环。

它不是单纯的论坛，也不是只有内容展示的官网，更不是通用 CMS。

## 2. 目标用户

- 学生用户：浏览竞赛、下载资料、报名参赛、查看审核结果、围绕竞赛提问
- 比赛管理员：维护自己负责的比赛、配置报名、审核报名、导出数据
- 内容编辑：发布通知、FAQ、附件，维护前台信息质量
- 全站管理员：配置平台、分配角色、管理全部比赛和用户

## 3. 核心业务主线

### 3.1 竞赛门户与通知展示

目标：让学生快速知道“有什么比赛、什么时候报名、去哪里下载资料”。

至少需要支持：

- 首页展示当前可报名比赛、最新通知、热门问答、资料下载入口
- 比赛列表页，按状态筛选：报名中、即将开始、进行中、已结束
- 比赛详情页，展示比赛介绍、时间安排、报名状态、通知公告、FAQ、附件下载、关联问答
- 通知公告按比赛归属管理，不能全部混在一起

### 3.2 比赛报名与审核

这是平台第一优先级的业务核心。

至少需要支持：

- 每个比赛配置自己的报名表
- 支持个人报名和团队报名
- 学生提交报名信息
- 限制重复报名
- 学生查看个人报名历史
- 管理员审核报名：通过、驳回、撤回、修改状态
- 记录审核日志
- 按比赛导出报名数据
- 按学院、专业、年级、状态统计报名情况

### 3.3 站内问答系统

问答是竞赛服务模块，不是独立社区。

至少需要支持：

- 学生提问
- 学生或管理员回答
- 问题绑定具体比赛
- 评论
- 点赞、收藏
- 采纳答案
- 标签分类
- 置顶、精华、隐藏、审核
- 在比赛详情页展示该比赛相关问答

### 3.4 用户、身份与权限

这是后台可控运行的基础。

至少需要 4 类角色：

- `super_admin`：全站管理员，管理所有比赛、用户、权限、配置
- `competition_admin`：比赛管理员，只管理自己负责的比赛
- `content_editor`：内容编辑，只能发布通知、FAQ、附件
- `student_user`：学生用户，可以报名、提问、回答、收藏

用户系统需要支持：

- 学生基础信息：姓名、学号、学院、专业、年级、邮箱、手机号
- 登录认证
- 第三方账号绑定预留，例如微信、学校统一认证
- 用户状态管理，例如正常、禁用、待验证
- 比赛级权限作用域，避免某个比赛管理员越权管理其他比赛

## 4. 后台管理范围

后台是业务管理台，不是简单 CMS。

需要支持：

- 比赛管理：创建比赛、编辑时间、设置状态、上传封面
- 通知管理：发布、撤回、排序、绑定比赛
- FAQ 管理：维护常见问题
- 附件管理：上传比赛文件、报名模板、通知材料
- 报名表管理：配置字段、必填项、选项、校验规则
- 报名审核：查看、筛选、批量处理、写审核意见
- 数据导出：导出 Excel/CSV
- 用户管理：查看用户、设置角色、禁用账号
- 问答管理：审核问题、隐藏回答、置顶问题、处理违规内容

## 4.1 技术路线约束

当前默认技术路线：

- `Next.js + TypeScript + Tailwind CSS + shadcn/ui`

当前默认模板策略：

- 主骨架使用 `shadcn-dashboard-landing-template-main/nextjs-version`
- 门户首页展示层参考 `launch-ui-main`

说明：

- 这里的“参考”是指吸收 section 组件和视觉语言，不是维护两套独立主站。
- 不优先选 `Ant Design Pro`，避免整体产品气质偏传统教务后台。

## 5. 第一版范围边界

### 必做

- 首页
- 比赛列表
- 比赛详情
- 登录
- 学生报名
- 我的报名
- 管理员创建比赛
- 管理员查看报名
- 管理员审核报名
- 报名数据导出

### 后置

- 完整问答系统
- 点赞、收藏、复杂标签
- 热门排行榜
- 微信登录、统一认证接入
- 精细统计看板
- 复杂 RBAC 配置界面

## 6. 关键业务对象

建议后续实现时至少围绕这些对象建模：

- `competition`：比赛主体
- `competition_notice`：比赛通知
- `competition_attachment`：比赛附件
- `competition_faq`：比赛 FAQ
- `registration_form`：报名表定义
- `registration`：报名记录
- `registration_team_member`：团队成员
- `registration_audit_log`：审核日志
- `question`：问题
- `answer`：回答
- `comment`：评论
- `favorite`：收藏
- `user`：用户
- `role_assignment`：角色与比赛作用域关联

## 7. 成功标准

第一阶段文档和实现应优先验证以下问题是否被解决：

- 学生能否在 3 分钟内完成“找到比赛并提交报名”
- 管理员能否在一个页面内完成“查看报名、筛选、审核、导出”
- 比赛信息是否按比赛聚合，而不是散落在公告、FAQ、附件多个孤立入口
- 平台结构是否明显体现“竞赛服务闭环”，而不是模板化后台或论坛首页
