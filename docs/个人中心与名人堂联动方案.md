# 个人中心与名人堂联动方案（待评审）

## 1. 背景

当前仓库已经具备以下基础：

- 已有登录态“我的报名”页面：`/me/applications`
- 已有公开态名人堂列表页：`/hall-of-fame`
- 已有公开态人物主页：`/profile/[id]`
- 已有 `users` 表基础字段：`name / email / image / studentNo / college / major / grade / phone`
- 已有名人堂与经验文章的 mock 数据结构：`hallOfFameEntries / experiencePosts`

但目前缺少一个统一的登录态“个人中心”入口，导致：

- 学生只能分散地查看“我的报名”，无法维护自己的个人资料
- 公开态名人堂主页与登录态本人资料没有明确的同步关系
- 经验文章虽然有公开展示模型，但没有登录态管理入口

本方案目标是在不偏离“学院竞赛管理与问答平台”定位的前提下，新增个人中心能力，并与现有名人堂模块形成闭环。

## 2. 设计原则

### 2.1 参考图只借结构，不照搬业务语义

用户提供的参考图是典型账户中心布局，可借鉴：

- 顶部站点导航
- 左侧纵向菜单
- 右侧主内容区
- “查看态 / 编辑态”切换
- 头像 + 基础资料表单

但当前项目不是电商或培训付费平台，因此不直接照搬：

- “我的订单”
- “我的发票”
- “常用发票抬头”

这些栏目应转译为竞赛平台语义。

### 2.2 优先最小可验证范围

本轮先形成可评审、可落地的个人中心方案，不直接扩散到无关系统。

优先级建议：

1. 我的信息
2. 我的报名
3. 我的成果 / 经验文章
4. 名人堂联动

### 2.3 明确区分登录态与公开态

- `/me/*`：登录后本人管理页
- `/profile/[id]`：公开展示页，仅对名人堂成员开放

个人中心负责“维护本人资料、管理公开内容、查看名人堂状态”。
公开人物页负责“展示经过筛选后的公开信息与经验文章”。

### 2.4 头衔与权限分离

本次新增的“头衔”应视为展示层身份系统，不直接替代权限模型。

- `role`：决定能做什么，例如是否可进入后台
- `title`：决定如何被展示，例如是否显示“管理员”“创世主”“百战小天才”

一个用户可以同时拥有多个头衔。

例如：

- 网站所有者账号：可同时拥有“创世主”与“管理员”
- 后台运营用户：可拥有“管理员”
- 高频参赛学生：可拥有“百战小天才”

## 3. 参考图转译方案

建议将参考图中的左侧菜单映射为：

| 参考图栏目 | 本项目对应栏目 | 说明 |
| --- | --- | --- |
| 我的信息 | 我的信息 | 维护个人基础资料与公开资料 |
| 我的订单 | 我的报名 | 复用现有报名记录与状态流转 |
| 我的发票 | 我的成果 | 管理竞赛成果、获奖信息、经验文章 |
| 常用发票抬头 | 名人堂展示 | 查看入选状态、公开页预览、展示开关 |

结论：

- 页面结构参考图保留
- 业务语义按竞赛平台重写
- 不引入订单、发票等无关概念

## 4. 路由建议

建议新增个人中心主路由组：

```text
/me
/me/profile
/me/applications
/me/achievements
/me/hall-of-fame
```

说明：

- `/me`：默认重定向到 `/me/profile`
- `/me/profile`：我的信息
- `/me/applications`：我的报名，保留现有页面并并入个人中心布局
- `/me/achievements`：我的成果与经验文章管理
- `/me/hall-of-fame`：名人堂联动页

这样做的好处是：

- 不推翻现有 `/me/applications`
- 个人中心形成稳定的路由分区
- 后续继续扩展账号安全、通知订阅时也有承载位置

### 4.1 阶段与数据源边界

为避免与 `docs/plan-hall-of-fame.md` 冲突，本方案明确分阶段处理数据源：

| 阶段 | 目标 | 数据源策略 |
| --- | --- | --- |
| 第一阶段 | `/me` 路由骨架、统一布局、基本资料查看/编辑壳子 | 不新增 `hall_of_fame` 相关真实表；公开态名人堂与公开人物页继续沿用现有 mock / 现有公开仓储 |
| 第二阶段 | 打通个人资料读写、公开预览、头像上传 | 只引入 `user_profile`；`users` 继续为学籍主数据源 |
| 第三阶段 | 成果与经验文章管理 | 引入真实 `experience_post`，并处理 mock `isPublished` 到状态枚举的迁移 |
| 第四阶段 | 后台治理、名人堂运营化 | 在后台 CRUD 就绪后，再考虑把 `hall_of_fame_entry` 从 mock 切到真实表 |

结论：

- 本方案不替代 `docs/plan-hall-of-fame.md`
- 第一阶段仍遵守“公开态名人堂暂不建真实表”的约束
- 第 7 节中的表结构属于后续真实 schema 预案，不代表第一阶段立即建表

### 4.2 登录守卫

`/me/*` 路由组应统一使用服务端登录守卫，而不是让每个页面各自兜底。

建议规则：

- 未登录访问 `/me/*`：重定向到 `/sign-in?callbackUrl=/me/...`
- `session.user.id` 缺失：同样视为不可进入个人中心，统一重定向到登录页

原因：

- 现有 `SessionUser.id` 是可选字段
- `/me/*` 中的资料、成果、名人堂状态查询都依赖稳定的当前用户 ID
- 比起在页面里到处写 `sessionUser.id ? ... : []`，统一守卫更不容易遗漏

## 5. 页面结构

### 5.1 整体布局

建议复用参考图的两栏结构：

```text
┌─────────────────────────────────────────────────────────────┐
│ PortalNavbar                                               │
├─────────────────────────────────────────────────────────────┤
│ 个人中心容器                                                │
│ ┌──────────────┬──────────────────────────────────────────┐ │
│ │ 左侧菜单      │ 右侧内容区                               │ │
│ │ 我的信息      │ 标题 + 编辑按钮                          │ │
│ │ 我的报名      │ 表单 / 列表 / 状态卡 / 预览区            │ │
│ │ 我的成果      │                                          │ │
│ │ 名人堂展示    │                                          │ │
│ └──────────────┴──────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

布局要求：

- PC 端：左 240px 左右固定导航，右侧自适应内容
- 移动端：统一采用顶部 Tabs，不再嵌套 Drawer，避免与现有移动端 navbar 的 `Sheet` 抽屉叠加
- 视觉上保持与当前门户页一致，不引入营销化后台特效
- `/me/applications` 纳入统一布局时，需要从原页面中移除 `PortalNavbar / PortalFooter / Section` 自包裹结构，避免重复渲染
- 第一阶段要把 `/me/applications` 视为一次受控迁移，而不是简单“包一层 layout”

### 5.2 我的信息页

目标：承接参考图中的表单编辑体验，但改造成竞赛平台资料页。

建议字段分层如下。

### A. 基础身份信息

- 头像
- 姓名
- 昵称
- 性别
- 出生日期

### B. 学籍与院系信息

- 学号
- 学校
- 学院
- 院系
- 专业
- 年级
- 入学年份
- 最高学历
- 是否在校

### C. 联系方式

- 邮箱
- 手机号

### D. 对外展示信息

- 个人简介
- 擅长方向 / 标签
- 头衔预览
- 是否允许公开展示头像
- 是否允许公开展示院系专业
- 是否允许公开展示头衔

说明：

- 其中 `姓名 / 学号 / 邮箱 / 学院 / 专业 / 年级 / 手机号` 直接复用真实 `users` 表已有字段，并作为该类信息的唯一主数据源
- `学校 / 院系 / 性别 / 出生日期 / 入学年份 / 最高学历 / 是否在校 / 昵称 / 对外展示开关` 建议放入新增扩展资料表
- 个人中心修改 `学院 / 专业 / 年级 / 手机号` 时，直接写回 `users`，不在 `user_profile` 中重复存储
- 报名表单默认继续读取 `users.college / users.major / users.grade / users.phone` 作为预填值
- 公开页中的“学院 / 专业”默认也从 `users` 读取，再叠加 `user_profile.department` 作为展示补充字段
- 头衔本身不允许用户自行编辑，只允许查看、预览和控制是否公开展示
- 页面默认是“只读态”，点击“编辑”后切换到表单态，符合参考图交互

建议补一张字段归属与读写规则表：

| 字段 | 主存储 | 写入方 | 报名预填来源 | 公开页来源 |
| --- | --- | --- | --- | --- |
| 姓名 | `users.name` | 用户本人 / 管理员 | 是 | 是 |
| 学号 | `users.studentNo` | 用户本人 / 管理员 | 是 | 否 |
| 学院 | `users.college` | 用户本人 / 管理员 | 是 | 是 |
| 专业 | `users.major` | 用户本人 / 管理员 | 是 | 是 |
| 年级 | `users.grade` | 用户本人 / 管理员 | 是 | 默认否 |
| 手机号 | `users.phone` | 用户本人 / 管理员 | 是 | 否 |
| 昵称 | `user_profile.nickname` | 用户本人 | 否 | 是 |
| 院系 | `user_profile.department` | 用户本人 | 否 | 是 |
| 学校 | `user_profile.schoolName` | 用户本人 | 否 | 可选 |
| 个人简介 | `user_profile.publicBio` | 用户本人 | 否 | 回退来源 |
| 头像公开开关 | `user_profile.publicShowAvatar` | 用户本人 | 否 | 是 |
| 头衔公开开关 | `user_profile.publicShowTitles` | 用户本人 | 否 | 是 |
| 擅长方向/标签 | `user_profile.skillTagsJson` | 用户本人 | 否 | 是 |

说明：

- 报名提交时写入 `registrations.applicantSnapshotJson` 的学籍快照，统一取自 `users`
- `user_profile` 不承担报名主数据源职责
- 这样可以避免 `users` 与 `user_profile` 双写同一学籍字段

### 5.3 我的报名页

目标：把当前独立存在的 `/me/applications` 纳入个人中心。

建议保留现有能力：

- 报名记录列表
- 状态时间线
- 审核意见
- 空状态提示

建议增强但不改变业务本质：

- 顶部增加个人中心统一页头
- 提供按比赛状态筛选
- 与“我的信息”页联动提示资料是否完整

示例：

- 若手机号、专业、年级为空，顶部提示“完善资料后可减少重复填写”

### 5.4 我的成果页

这是本次与名人堂联动的关键过渡层。

建议包含两块内容：

### A. 我的竞赛成果

- 关联比赛
- 获奖等级
- 获奖年份
- 团队 / 个人
- 是否已用于名人堂展示

### B. 我的经验文章

- 文章标题
- 关联比赛
- 发布状态：草稿 / 待审核 / 已发布 / 已下线

- 操作：编辑 / 预览 / 提交发布 / 下线

说明：

- 公开人物页里展示的“经验文章”来源于这里
- 个人中心是作者管理入口，`/profile/[id]` 是公开消费入口

### 5.5 名人堂展示页

这是个人中心与名人堂的直接联动页，建议独立出来，不混在“我的信息”页末尾。

建议包含 4 个模块：

### A. 入选状态卡

状态建议：

- 未入选
- 候选中
- 已入选但资料未完善
- 已入选并公开展示
- 已下线

### B. 公开信息预览

预览内容对齐公开人物页：

- 头像
- 姓名 / 昵称
- 头衔
- 擅长方向/标签
- 学院 / 院系 / 专业
- 名人堂标签
- 个人简介
- 已发布经验文章列表

### C. 展示设置

- 是否公开展示头像
- 是否公开展示院系专业
- 是否启用个人简介
- 是否启用经验文章展示

### D. 跳转动作

- 前往公开主页 `/profile/[id]`
- 前往名人堂列表 `/hall-of-fame`
- 前往我的成果，继续维护经验文章

## 6. 与名人堂的联动规则

### 6.1 公开页不直接读取全部个人中心字段

公开人物页 `/profile/[id]` 只读取允许公开的字段，不直接把个人中心整页内容搬到前台。

建议公开字段来源如下：

| 公开区块 | 数据来源 | 说明 |
| --- | --- | --- |
| 头像 | `users.image` 或 `user_profile.avatar` | 需受展示开关控制 |
| 展示姓名 | `nickname` 优先，其次 `name` | 允许个人中心维护 |
| 头衔 | `resolvedUserTitles` | 由系统解析后返回，需受公开开关控制 |
| 学院/专业 | `users.college / users.major` | 作为学籍信息主数据源，受公开开关控制 |
| 院系 | `user_profile.department` | 作为展示补充字段，非报名主数据 |
| 名人堂标签 | `hall_of_fame_entry.tag` | 建议仍由管理员维护 |
| 名人堂简介 | `hall_of_fame_entry.adminBio ?? user_profile.publicBio` | 学生撰写，管理员可覆盖 |
| 经验文章 | `experience_post` 已发布记录 | 从“我的成果”发布出来 |

### 6.2 推荐的数据主从关系

建议采用以下主从关系，避免多人同时维护同一内容：

- `hall_of_fame_entry`
  - 管理员维护：入选状态、标签、排序、是否公开、`adminBio`
- `user_profile`
  - 用户本人维护：头像、昵称、`publicBio`、公开开关、补充资料
- `experience_post`
  - 用户本人撰写，管理员可审核后公开

这样可以实现：

- 管理员控制“谁能上名人堂”
- 学生控制“公开后展示什么细节”
- 经验文章成为公开人物页的内容来源

其中 `bio` 建议采用“学生撰写 + 管理员覆盖”策略：

- 学生在个人中心维护 `user_profile.publicBio`
- 管理员在名人堂后台可选填写 `hall_of_fame_entry.adminBio`
- 公开页优先显示 `adminBio`
- 当 `adminBio` 为空时，回退到 `publicBio`

### 6.3 个人中心到名人堂的触发逻辑

建议规则：

1. 用户未入选名人堂时：
   - 可以维护个人资料
   - 可以写经验文章草稿
   - 不生成公开人物页入口

2. 用户被管理员设为名人堂成员时：
   - 个人中心出现“名人堂展示”菜单
   - 若公开资料不完整，显示“待完善”状态
   - 完善后可生成公开态展示

3. 用户的经验文章发布成功后：
   - 自动出现在 `/profile/[id]`
   - 可被名人堂人物页和名人堂详情消费

### 6.4 头衔规则

建议首版内置以下 3 个头衔：

| 头衔 | 类型 | 触发规则 | 建议优先级 | 备注 |
| --- | --- | --- | --- | --- |
| 创世主 | 固定账号 | 绑定网站所有者个人账号 | 100 | 全站唯一，最高优先级 |
| 管理员 | 固定账号 | 绑定预设管理员账号 | 80 | 不按角色自动推导 |
| 百战小天才 | 统计计算 | 累计已提交报名的不同比赛数默认 `> 10` | 60 | 首版按需求实现，阈值保留配置化入口 |

#### 创世主

- 用途：标识网站所有者的个人账号
- 你已确认：`创世主` 采用固定账号绑定
- 建议实现：通过固定 `ownerUserId` 配置绑定
- 特性：
  - 全站唯一
  - 不允许普通后台用户修改
  - 在所有展示场景中优先显示

#### 管理员

- 用途：标识具备后台管理权限的用户
- 你已确认：`管理员` 也采用固定账号绑定
- 建议实现：通过固定 `adminUserIds` 配置维护
- 特性：
  - 与后台权限是两套概念
  - 不按 `role` 自动推导
  - 可以有后台权限但不展示“管理员”头衔
  - 也可以存在“管理员”头衔但后台权限已经被撤销，因此实现上应额外做一致性检查或运营约束
  - 可作为登录态用户昵称旁的身份徽标

#### 百战小天才

- 用途：标识高活跃参赛用户
- 建议统计口径：
  - 统计 `registrations` 中当前用户 `applicantUserId` 对应的“已正式提交过”报名记录数
  - 计入状态：`submitted / approved / rejected / withdrawn`
  - 不计入状态：`draft / cancelled`
- 触发条件：
  - 首版默认按“报名超过 10 场比赛”授予
  - 阈值建议抽成配置常量，例如 `battleGeniusMinCompetitions`
- 设计原因：
  - 现有 `registration_competition_applicant_active_unique` 已保证 `(competitionId, applicantUserId)` 在计入状态内最多只有一条活跃记录
  - 因此服务层可直接按 `applicantUserId + status` 计数，不需要额外 `DISTINCT competitionId`
  - 避免同一比赛反复草稿或重复修改刷头衔
  - 更贴近“报名超过 10 场比赛”的业务含义

#### 展示规则

- 紧凑场景只展示 1 个主头衔：
  - 顶部导航用户区
  - 名人堂卡片
  - 公开人物页列表卡片
- 详情场景展示全部头衔：
  - 个人中心“我的信息”
  - 个人中心“名人堂展示”
  - `/profile/[id]` 人物主页
- 多头衔排序建议：
  - `创世主`
  - `管理员`
  - `百战小天才`

#### 与名人堂的关系

- 头衔不等于名人堂标签
- 名人堂标签偏向“代表性荣誉”或“个人定位”
- 头衔偏向“平台身份”与“活跃度勋章”

示例：

- 某用户可同时展示：
  - 头衔：`百战小天才`
  - 名人堂标签：`国赛一等奖`

## 7. 数据模型建议

说明：

- 本节是后续真实 schema 预案
- 第一阶段不要求全部落表
- 其中 `user_profile` 早于 `hall_of_fame_entry` 落地，`hall_of_fame_entry` 需等后台治理阶段再决定是否从 mock 切换

### 7.1 复用现有 `users` 表字段

### 7.2 新增扩展资料表 `user_profile`

建议新增扩展表承载参考图里的补充字段，避免把所有展示类字段都堆进 `users` 主表。

注意：

- `user_profile` 不再重复存储 `college / major / grade / phone`
- 学籍主数据保留在 `users`
- `user_profile` 只承载补充资料与公开展示控制

### 7.3 名人堂表建议

当前项目已有 mock `hallOfFameEntries`。后续转真实表时，建议至少包含：

```ts
hallOfFameEntry {
  id: uuid
  userId: uuid
  tag: string
  status: "candidate" | "active" | "hidden"
  displayOrder: number
  adminBio: text | null
  adminNote: string | null
  createdAt
  updatedAt
}
```

页面上的 5 类状态卡建议视为 UI 派生态，而不是数据库枚举直接一一对应：

| UI 状态 | 派生条件 |
| --- | --- |
| 未入选 | 无 `hall_of_fame_entry` |
| 候选中 | `status = candidate` |
| 已入选但资料未完善 | `status = active` 且公开资料不完整 |
| 已入选并公开展示 | `status = active` 且公开资料完整 |
| 已下线 | `status = hidden` |

### 7.4 经验文章表建议

当前项目已有 mock `experiencePosts`。后续转真实表时建议字段：

```ts
experiencePost {
  id: uuid
  userId: uuid
  competitionId: uuid | null
  title: string
  contentHtml: text
  coverImage: string | null
  awardLevel: string | null
  status: "draft" | "pending_review" | "published" | "offline"
  publishedAt: timestamp | null
  createdAt
  updatedAt
}
```

迁移注意：

- 现有 mock 结构使用 `isPublished: boolean`
- 现有公开消费逻辑按 `isPublished` 过滤
- 第三阶段切换到真实 `experience_post.status` 后，需要同步修改：
  - `ExperiencePost` 类型定义
  - `profile-repository` 公开过滤逻辑
  - 个人中心成果页状态展示
  - 可能的 seed / mock 兼容层

这是一个明确的 breaking change，应作为第三阶段迁移点单独处理

### 7.5 头衔模型建议

首版建议简化，不单独建头衔表，直接采用“配置 + 服务层解析”方案。

#### A. 配置文件定义头衔元数据

建议在 `src/lib/constants/user-titles.ts` 一类文件中定义：

```ts
export const USER_TITLE_DEFINITIONS = {
  founder: { label: "创世主", priority: 100, sourceType: "fixed" },
  admin: { label: "管理员", priority: 80, sourceType: "fixed" },
  battleGenius: { label: "百战小天才", priority: 60, sourceType: "computed" },
} as const;
```

#### B. 服务层解析结果 `resolvedUserTitles`

建议由 `user-title-service` 统一合并：

- 固定账号头衔：
  - `创世主`：来自固定 `ownerUserId`
  - `管理员`：来自固定 `adminUserIds`
- 统计头衔：
  - `百战小天才`：来自 `registrations` 聚合结果

这样可以减少状态不一致问题，也避免首版为 3 个头衔过早引入额外表结构。

#### C. 后续扩展时机

当满足以下任一条件时，再考虑引入 `title_definition / user_title_grant` 正式表：

- 头衔种类超过 5-6 个
- 需要后台动态管理头衔
- 需要按时间授予、撤销或人工审批头衔
- 需要活动型、限时型头衔

## 11. 验收标准

满足以下条件可认为第一版个人中心方向正确：

1. 登录用户可以从统一入口进入个人中心，而不是只看到“我的报名”
2. 用户可以在“我的信息”页维护基础资料，并区分私有字段与公开字段
3. 用户可以在个人中心查看自己的名人堂状态
4. 用户可以在个人中心查看自己已获得的头衔与公开展示效果
5. 已入选名人堂的用户可以预览公开人物页
6. 公开人物页仍严格遵守隐私边界，不暴露报名审核数据
7. 经验文章的管理入口与公开展示来源一致，不出现两套内容源

## 12. 本轮建议结论

建议将“个人中心”定义为：

**学生登录后的资料管理与公开展示控制枢纽，而不是简单的报名查询页。**

它与名人堂的关系应当是：

- 个人中心负责维护本人资料与公开内容
- 名人堂负责公共曝光与荣誉展示
- `/profile/[id]` 是公开结果页

也就是说，名人堂不是个人中心里的一个孤立页面，而是个人中心中“公开身份”能力的外显结果。

## 13. 待你确认的问题

在进入实际编码前，建议先确认这 5 点：

1. 左侧栏目是否采用 `我的信息 / 我的报名 / 我的成果 / 名人堂展示` 这一版命名
2. “我的成果”是否需要首版就包含经验文章编辑，还是先只做列表与占位
3. 名人堂简介 `bio` 的最终维护方是学生本人、管理员，还是“双轨制”
4. 固定“管理员”账号是一人还是多人名单
5. `/me` 是否作为统一个人中心首页，还是继续只保留 `/me/applications` 并逐步扩展
