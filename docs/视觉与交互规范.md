# 视觉与交互规范

## 1. 设计目标

这个平台的视觉风格应体现：

- 学院官方感
- 竞赛服务感
- 清晰、可信、可操作

不应体现：

- 泛社区论坛感
- 通用后台模板拼装感
- 炫技但弱业务的运营页风格

## 2. 整体视觉原则

### 信息优先

- 重要信息是比赛名称、时间、状态、报名入口、审核结果，不是装饰图形。
- 页面应优先服务“读懂和操作”，再考虑风格强化。

### 页面角色明确

- 前台页面偏公开信息服务
- 报名页偏表单任务完成
- 审核页偏高密度业务操作

同一套组件可以复用，但不能把三类页面都做成同一种模板节奏。

### 统一状态语言

- 比赛状态、报名状态、审核状态必须有统一颜色和文案
- 同一状态在首页、详情页、后台表格里的呈现不能冲突

### 设计令牌初稿

状态颜色建议先固定为语义色，不在不同页面随意漂移：

| 语义 | 默认色系 |
|:---|:---|
| `draft` | `slate` |
| `upcoming` | `sky` |
| `registration_open` | `emerald` |
| `in_progress` | `indigo` |
| `finished` | `zinc` |
| `archived` | `gray` |
| `submitted / pending_review` | `amber` |
| `approved / published` | `emerald` |
| `rejected` | `rose` |
| `withdrawn` | `orange` |
| `cancelled / hidden / disabled` | `slate` |

基础版式令牌建议如下：

- 字体层级：`12 / 14 / 16 / 20 / 24 / 32`
- 间距基准：`4 / 8 / 12 / 16 / 24 / 32 / 48`
- 圆角基准：`8 / 12 / 16`
- 卡片阴影：公开页用轻阴影，后台表格页优先弱阴影或无阴影

实现约束：

- 后续应在 CSS 变量或主题 token 中落地这些语义色
- 业务状态标签优先使用“语义 token”，不要在页面里手写零散颜色类

## 3. 页面风格建议

### 首页

- 风格要像“学院竞赛门户”，不是 SaaS 营销页
- 头图可简洁，但必须立即说明平台价值和当前重点比赛
- 重点模块应是报名中比赛、最新通知、资料下载，而不是空泛卖点区
- 可以参考 `Launch UI` 的 `hero`、`stats`、`faq`、`cta` 组件节奏，但内容必须围绕竞赛业务重写

### 比赛列表

- 以卡片或紧凑列表承载信息
- 每张卡片至少突出：比赛名、时间、状态、报名动作
- 筛选器应固定在上方，减少来回跳转

### 比赛详情

- 首屏摘要区必须强，学生进入后应立刻知道“能不能报、什么时候报、需要看什么材料”
- 通知、FAQ、附件、问答建议采用分段结构或锚点导航
- 不要把长文案全部堆成一个页面大段落

### 报名页

- 表单应分组展示，例如基础信息、参赛信息、团队成员、附件上传
- 必填、校验失败、重复报名等反馈必须明显
- 提交按钮附近要明确告知审核后续

### 报名审核页

- 优先使用强表格布局
- 筛选区、表格区、详情抽屉区层次要清楚
- 审核动作应稳定放在固定区域，避免误操作
- 可以优先复用 `ShadcnStore Dashboard` 现成的 table、tasks、users 页结构，再替换为报名审核语义

## 4. 内容组件建议

建议优先定义并复用以下组件：

- 比赛状态标签
- 报名状态标签
- 通知列表卡片
- 附件下载列表
- FAQ 折叠项
- 报名表字段渲染器
- 审核意见面板
- 审核日志时间线

## 5. 文案与交互规则

- 按钮文案使用明确动作，例如“立即报名”“查看通知”“提交审核”“导出报名数据”
- 空状态不要只写“暂无数据”，应说明下一步动作
- 驳回、撤回、取消等高风险动作必须二次确认
- 审核结果页应给出明确原因，不要只有状态变化

## 6. 响应式要求

- 首页、比赛列表、比赛详情必须兼容移动端
- 报名表在移动端可以分步或折叠，但不能因布局导致字段难以填写
- 后台审核页优先保证桌面端效率，移动端至少要可读，不强求完整高效操作

## 7. 前端实现提醒

- 先做 Mock 数据版前端，用真实字段名和状态名占位，不要用泛化示例数据
- 不要先接一堆图表；审核页首要任务是处理记录，不是展示 dashboard
- 如果后续基于现成模板改造，必须先替换默认页面结构和文案，再谈样式细化
- `Launch UI` 用于首页展示层补强，`ShadcnStore` 用于整体骨架与后台，不要把两边都当成完整站点并行维护
