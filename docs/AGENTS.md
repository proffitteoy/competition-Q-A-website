## 1. 项目定位

本项目当前应统一理解为：

**学院竞赛管理与问答平台**

不是泛论坛，不是普通官网，不是通用 CMS。

## 3. 文档修改原则

- 优先做最小必要修改，不引入与当前目标无关的大段模板内容。
- 先把业务闭环写清楚，再谈抽象架构或通用能力。
- 文档内容必须体现优先级：`报名与审核` 第一，`竞赛门户` 第二，`问答沉淀` 第三。
- 如果文档与现有模板内容冲突，以当前项目文档为准，不再回退到“通用模板”表述。

## 4. 需求收敛规则

### 第一版必须围绕以下问题展开

- 学生如何发现比赛并完成报名
- 管理员如何查看、审核、导出报名
- 比赛信息如何按比赛聚合展示
- 问答如何服务竞赛，而不是变成独立社区

### 第一版不应优先展开

- 完整社区化运营功能
- 复杂统计大盘
- 过细的权限配置界面
- 与 MVP 主链路无关的视觉炫技

## 5. 如果后续进入实现阶段

- 先做 Mock 数据版前端，验证页面结构和交互，不要直接铺后端全量实现。
- 若开始改代码，应先确认真正的项目入口目录，不要误改当前仓库中的第三方模板或上游源码副本。
- 只有在明确要落地某个实现目录时，才补具体技术栈、命令和目录边界文档。

## 6. 输出要求

每次更新文档后，应尽量说明：

- 改了哪些文档
- 收敛了哪些范围
- 哪些能力明确纳入 MVP
- 哪些能力被后置
